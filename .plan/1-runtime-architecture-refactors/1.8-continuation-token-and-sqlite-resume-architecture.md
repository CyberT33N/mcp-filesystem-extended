# Task 1.8: Continuation token and SQLite resume architecture
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Implement the same-endpoint continuation architecture for the affected inspection families so preview-first or task-backed-required outcomes can return a server-issued continuation token, persist the normalized query and continuation state in local SQLite storage, and resume later through the same public endpoint without forcing the agent to resend the original query.

This task is the final runtime architecture layer for the preview / task-backed workflow discussed in the current analysis.

## Expected Result
[INTENT: KONTEXT]

- The affected endpoint families accept a continuation-only resume request that carries `continuationToken`.
- The server persists the normalized request and continuation state in one SQLite-backed local store.
- Preview-capable families return preview data plus continuation metadata.
- `count_lines` returns task-backed continuation metadata without partial preview totals.
- Invalid continuation tokens are handled server-side through canonical not-found-class behavior.
- `read_file_content` and `read_files_with_line_numbers` remain outside this continuation-token architecture.

## Preconditions
[INTENT: CONSTRAINT]

- Task `1.7` must be completed.
- The execute agent must not introduce an agent-controlled `force full` or `disable preview` request flag.
- The execute agent must not introduce a separate public big-search or big-traversal endpoint.
- The execute agent must not add a new external SQLite package because builtin `node:sqlite` is already available via the current Node runtime requirement.

## External Runtime Reference
[INTENT: REFERENZ]

Use the user-supplied Node.js builtin SQLite documentation as the authoritative storage API reference for this task.

The required surfaces are:

- `node:sqlite`
- `DatabaseSync`
- `database.exec()`
- `database.prepare()`
- `statement.run()`
- `statement.get()`
- `statement.all()`
- optional `database.createTagStore()` for prepared-statement caching

The task must stay on the builtin Node runtime surface and must not introduce a new external SQLite client.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Same-endpoint continuation request contract
**Type:** `constraint`

**Description:**
Affected endpoint families must support one continuation-only request mode that carries `continuationToken` and no new query-defining fields.

**Ist-Zustand:**
The affected endpoint schemas accept only the base request contract and have no additive resume token surface.

**Soll-Zustand:**
The affected schemas accept either the original base request or a continuation-only request carrying `continuationToken`; the original query is loaded from persistence rather than resent by the agent.

**Architectural Rationale:**
The agent should not need to repeat or reconstruct the original query once the server has already admitted and persisted the large workload.

### IU-002: SQLite-backed local continuation store
**Type:** `object`

**Description:**
The server must persist the normalized original request, the continuation state, lifecycle timestamps, and the owning endpoint family in one local SQLite store.

**Ist-Zustand:**
There is no current persistent continuation store, so no server-issued token can survive process restarts or longer IDE pauses.

**Soll-Zustand:**
One builtin-`node:sqlite` store persists continuation rows in `inspection_continuation_sessions` and keeps active tokens resumable across process restarts within the sliding TTL window.

**Architectural Rationale:**
Pure in-memory storage is insufficient for the long-running IDE and return-later workflow the architecture must support.

### IU-003: Preview-capable family response workflow
**Type:** `workflow`

**Description:**
Regex, fixed-string, and discovery families must return preview data plus continuation metadata when preview-first admission is active.

**Ist-Zustand:**
The families can already emit preview-like truncation behavior or narrowing guidance, but they do not yet return a resumable token or a shared continuation contract.

**Soll-Zustand:**
The response includes existing data fields plus additive authoritative `admission` and `continuation` objects that tell the agent whether more data is available and how to resume.

**Architectural Rationale:**
Preview is only useful as a complete workflow when the same endpoint family can later continue the request deterministically.

### IU-004: Count-lines task-backed workflow
**Type:** `workflow`

**Description:**
`count_lines` must become continuation-capable without introducing partial preview totals.

**Ist-Zustand:**
`count_lines` currently has no continuation token and no task-backed full-resume contract.

**Soll-Zustand:**
Once `count_lines` leaves the inline band, it returns continuation metadata and later resumes through the same endpoint family, but it does not emit partial preview totals.

**Architectural Rationale:**
Partial line totals are semantically misleading; full continuation is the correct behavior once inline execution is no longer appropriate.

### IU-005: Error behavior, retention, and cleanup
**Type:** `constraint`

**Description:**
The continuation architecture must define deterministic error semantics and a durable retention model.

**Ist-Zustand:**
There is no continuation-token lifecycle, so there is also no canonical invalid-token handling or cleanup policy.

**Soll-Zustand:**
Invalid or unusable tokens collapse into one not-found-class continuation failure; continuation requests that include new query fields fail invalid-request validation; active rows use a 30-day sliding TTL; non-active rows clean out after 7 days.

**Architectural Rationale:**
The system must stay resumable for realistic IDE pauses while still enforcing bounded local state growth and safe token semantics.
