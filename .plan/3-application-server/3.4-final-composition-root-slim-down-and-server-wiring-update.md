# Task 3.4: Final composition root slim-down and server wiring update
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Replace the oversized monolithic application catalog body with a bounded composition root that delegates to the extracted registration modules.
Align the application server wiring only as much as necessary to support the decomposed registration structure while preserving existing MCP server behavior.

## Expected Result
[INTENT: KONTEXT]
- [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) becomes a thin composition root that delegates to the extracted registration modules.
- The file no longer owns domain output schemas, inline preset constants, or server-scope registration details.
- [`filesystem-server.ts`](src/application/server/filesystem-server.ts) remains behaviorally stable and only changes if the new composition contract makes a small wiring adjustment necessary.

## Preconditions
[INTENT: CONSTRAINT]
- Tasks `3.1`, `3.2`, and `3.3` are complete.
- All newly created application registration and support modules have been re-read immediately before execution.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Delegated application composition root
**Type:** `object`

**Description:**
The monolithic application catalog is replaced by a slim composition root that delegates to bounded registration modules.

**Ist-Zustand:**
[`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) contains inline registration for all tool families, inline preset constants, and inline server-scope tool registration.

**Soll-Zustand:**
[`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) retains only the shared registration context type and the orchestration logic that calls the extracted bounded registration modules.

**Architectural Rationale:**
This is the application-layer equivalent of bounded context decomposition. The root file should orchestrate, not own every registration detail.

**Exact Target:**
`src/application/server/register-tool-catalog.ts`

**Implementation Contract:**
Remove inline preset ownership, inline output schema ownership, and inline family-specific registration blocks. Delegate to the extracted modules instead.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `Slim delegated composition root` |
| Full Target Shape / Value | `Root file exports the registration context type and composes bounded registration modules only` |
| Authoritative Source | Current root file plus the extracted modules created in tasks `3.1`, `3.2`, and `3.3` |
| Determination Rule | Preserve behavior, remove inline ownership, delegate by bounded module |
| Exact Placement | `src/application/server/register-tool-catalog.ts` |
| Consumers | `filesystem-server.ts` |
| No-Breaking Constraint | Tool surface behavior and registration semantics remain unchanged |

### IU-002: Filesystem server wiring stability
**Type:** `constraint`

**Description:**
The filesystem server shell must remain behaviorally stable after the composition root slim-down.

**Ist-Zustand:**
[`filesystem-server.ts`](src/application/server/filesystem-server.ts) constructs the MCP server and calls `registerToolCatalog()` directly.

**Soll-Zustand:**
`filesystem-server.ts` keeps the same connection behavior, logging behavior, description/instruction wiring, and registration timing. Only minimal signature or import adjustments are allowed if the slimmed root requires them.

**Architectural Rationale:**
The server shell is the transport composition boundary. It should not absorb a redesign during modularization.

**Exact Target:**
`src/application/server/filesystem-server.ts`

**Implementation Contract:**
Do not change constructor semantics, connection flow, logging semantics, or MCP capability setup.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `Filesystem server shell wiring` |
| Full Target Shape / Value | `Behaviorally unchanged shell with only minimal import or call-shape alignment if required` |
| Authoritative Source | Current shell file and the final composition contract |
| Determination Rule | Prefer zero change unless the slimmed root requires a minimal wiring adjustment |
| Exact Placement | `src/application/server/filesystem-server.ts` |
| Consumers | Application entrypoint and MCP runtime |
| No-Breaking Constraint | Initialization and runtime behavior stay unchanged |

## Actions
[INTENT: ANWEISUNG]
1. **Re-read the extracted application modules and current root files**
   - **What:** Re-anchor the current root files and every extracted registration/support module from tasks `3.1`, `3.2`, and `3.3`.
   - **Where:** All files listed in `files_read`.
   - **How:** Verify the real current module interfaces before editing the root files.
   - **Why:** This task sits on the final shared application frontier and must not use stale assumptions.
   - **Inputs / References:** All authoritative references listed above.
   - **Output Contract:** A verified current-state interface map for all delegates and the two root files.
   - **Verification:** Every delegate import and expected registration context field is confirmed from current file contents.
   - **Stop if:** Any extracted module interface differs materially from the assumptions in this task file.
2. **Slim the composition root into a pure delegator**
   - **What:** Rewrite [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) so it composes the extracted bounded modules instead of owning inline registration detail.
   - **Where:** [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts)
   - **How:** Keep the shared context type if it is still needed, import the extracted bounded registration modules, and call them in the correct order.
   - **Why:** This completes the application-layer modularization.
   - **Inputs / References:** Current root file plus the extracted modules from tasks `3.1`, `3.2`, and `3.3`.
   - **Output Contract:** The root file is thin, bounded, and free of inline domain contract duplication.
   - **Verification:** No inline preset constants, no inline output schemas, and no inline family-specific registration blocks remain.
   - **Stop if:** The rewrite would require reintroducing compatibility shims or changing tool semantics.
3. **Align filesystem-server wiring only if necessary**
   - **What:** Apply only the minimal wiring update needed to keep the shell aligned with the slimmed composition root.
   - **Where:** [`filesystem-server.ts`](src/application/server/filesystem-server.ts)
   - **How:** Update imports or call signatures only if the composition root contract requires it; otherwise leave the file unchanged.
   - **Why:** The shell must remain stable.
   - **Inputs / References:** [`filesystem-server.ts`](src/application/server/filesystem-server.ts), [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts)
   - **Output Contract:** The shell still initializes and registers tools exactly as before from a behavior perspective.
   - **Verification:** Constructor and `connect()` flow remain unchanged except for minimal required wiring.
   - **Stop if:** The wiring update would alter server initialization semantics.
