[INTENT: KONTEXT]

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Exact Schema Changes for Line-Range Replacement
**Type:** `schema`

**Description:**
Apply the exact finite request caps to the line-range replacement schema.

**Ist-Zustand:**
The schema leaves the `files` array, `replacements` array, path strings, and replacement text effectively unbounded apart from line-number minima.

**Soll-Zustand:**
Apply these exact schema changes:

| Field | Value |
|------|-------|
| `files` | `.max(MAX_CONTENT_FILES_PER_REQUEST)` |
| each `path` | `.max(PATH_MAX_CHARS)` |
| `replacements` | `.max(MAX_REPLACEMENTS_PER_FILE)` |
| each `replacementText` | `.max(REPLACEMENT_TEXT_MAX_CHARS)` |
| `startLine` / `endLine` | keep existing integer/min constraints unchanged |

**Architectural Rationale:**
This preserves the endpoint contract while preventing unbounded replacement surfaces before the handler runs.

**Exact Target:**
`src/domain/mutation/replace-file-line-ranges/schema.ts`

**Implementation Contract:**
Import shared constants; do not inline numbers locally.

### IU-002: Exact Runtime Budget Policy for Replacement Handlers
**Type:** `constraint`

**Description:**
Define the cumulative-input and preview-output controls for the replacement handler/helper pair.

**Ist-Zustand:**
The helper reads the file, applies all replacements, and emits full unified diffs without any cumulative raw-text or preview-size budget enforcement.

**Soll-Zustand:**
Apply these exact runtime rules:

| Runtime Rule | Exact Requirement |
|------|-------|
| Cumulative replacement input | Sum all `replacementText.length` values across all files and refuse if the total exceeds `LINE_REPLACEMENT_TOTAL_INPUT_CHARS` before any write occurs |
| Preview/diff output | After the diff preview is built, refuse if the result exceeds `FILE_DIFF_RESPONSE_CAP_CHARS` |
| Dry-run behavior | Keep existing dry-run semantics unchanged; only add the new refusal paths |

**Architectural Rationale:**
The endpoint must remain deterministic and direct, but it cannot allow unbounded replacement text or oversized diff previews to leave the server.

**Exact Target:**
`src/domain/mutation/replace-file-line-ranges/handler.ts`, `src/domain/mutation/replace-file-line-ranges/helpers.ts`

**Implementation Contract:**
No truncation is allowed for successful preview output; oversize preview/diff output must be refused.

## Actions
[INTENT: ANWEISUNG]
1. **Update the replacement schema with the exact shared caps**
   - **What:** Apply the schema constraints from IU-001.
   - **Where:** `src/domain/mutation/replace-file-line-ranges/schema.ts`
   - **How:** Import the shared constants and add `.max(...)` to `files`, `path`, `replacements`, and `replacementText`.
   - **Why:** The request surface must be finite before the handler runs.
   - **Inputs / References:** shared limit matrix, current schema.
   - **Output Contract:** Same property names, same line-number semantics, but bounded request surfaces.
   - **Verification:** Confirm the schema still preserves existing line-number validation while bounding file count, replacement count, and replacement-text size.
   - **Stop if:** Implementing the caps would require renaming existing request fields.
2. **Add cumulative replacement-input enforcement**
   - **What:** Update the handler and/or helper to refuse oversize cumulative replacement text before any write occurs.
   - **Where:** `src/domain/mutation/replace-file-line-ranges/handler.ts` and/or `helpers.ts`
   - **How:** Sum replacement-text sizes across the request and compare against `LINE_REPLACEMENT_TOTAL_INPUT_CHARS` before the first write operation.
   - **Why:** Multiple individually valid replacements can still create an oversized mutation payload cumulatively.
   - **Inputs / References:** shared limit matrix, shared error contract.
   - **Output Contract:** No write occurs when the cumulative replacement budget fails.
   - **Verification:** Confirm the cumulative budget is checked before any file write is performed.
   - **Stop if:** Enforcing the cumulative budget would require silent truncation.
3. **Add preview/diff output budget enforcement**
   - **What:** Refuse oversize diff/preview output after it is built.
   - **Where:** `src/domain/mutation/replace-file-line-ranges/helpers.ts`
   - **How:** Use the shared text-budget helper to assert the final preview size against `FILE_DIFF_RESPONSE_CAP_CHARS`.
   - **Why:** Preview text can itself become too large even when the request payload is within bounds.
   - **Inputs / References:** shared text-budget helper, shared limits, shared error contract.
   - **Output Contract:** Successful preview output remains complete and untruncated; oversize preview output is refused.
   - **Verification:** Confirm the preview/diff result is budget-checked before success is returned.
   - **Stop if:** The preview budget would require removing existing fields or altering dry-run semantics.
