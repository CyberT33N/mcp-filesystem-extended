# Task 2.1: Inspection Metadata, Discovery, Count, and Checksum Guardrails
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Apply the shared guardrail contract to all non-read, non-regex inspection endpoints so they expose finite request surfaces and bounded result behavior without introducing breaking changes. This task hardens recursive listing, metadata lookup, name discovery, glob discovery, line counting, checksum generation, and checksum verification.

## Context
[INTENT: KONTEXT]
- **Domain:** Inspection-family contract and handler hardening for metadata-heavy and discovery-heavy endpoints.
- **Related Files:** `src/domain/inspection/*/{schema.ts,handler.ts,helpers.ts}`, shared guardrail modules.
- **Technical Context:** These endpoints mostly return path lists, metadata tables, checksum tables, or counters. Their risk is lower than direct reads and regex search, but their batch size and response fan-out still need finite ceilings.
- **Business Context:** The server must preserve legitimate wide-but-safe discovery requests while stopping runaway batches and overly large summary payloads.
- **Architectural Context:** This task should prefer schema caps first and only add handler truncation/refusal where the response fan-out remains dynamic after schema validation.

### IU-001: Exact Schema Guardrails for Inspection Non-Read Surfaces
**Type:** `schema`

**Description:**
Apply the exact shared limits to the targeted inspection schemas.

**Ist-Zustand:**
The targeted schemas use very small `min(1)` validation only and leave arrays/strings effectively unbounded.

**Soll-Zustand:**
Apply these exact additions:

| Target Schema | Required Changes |
|------|-------|
| `ListDirectoryEntriesArgsSchema` | `roots.max(MAX_DISCOVERY_ROOTS_PER_REQUEST)`, each root `.max(PATH_MAX_CHARS)`, `excludeGlobs.max(MAX_EXCLUDE_GLOBS_PER_REQUEST)`, each glob `.max(GLOB_PATTERN_MAX_CHARS)` |
| `GetPathMetadataArgsSchema` | `paths.max(MAX_GENERIC_PATHS_PER_REQUEST)`, each path `.max(PATH_MAX_CHARS)` |
| `FindPathsByNameArgsSchema` | `roots.max(MAX_DISCOVERY_ROOTS_PER_REQUEST)`, each root `.max(PATH_MAX_CHARS)`, `nameContains.max(LABEL_MAX_CHARS)`, `excludeGlobs` limits as above, optional `maxResults.max(500)` |
| `FindFilesByGlobArgsSchema` | `roots.max(MAX_DISCOVERY_ROOTS_PER_REQUEST)`, each root `.max(PATH_MAX_CHARS)`, `glob.max(GLOB_PATTERN_MAX_CHARS)`, `excludeGlobs` limits as above, `maxResults.max(500)` |
| `CountLinesArgsSchema` | `paths.max(MAX_GENERIC_PATHS_PER_REQUEST)`, each path `.max(PATH_MAX_CHARS)`, optional `regex.max(REGEX_PATTERN_MAX_CHARS)`, `includeGlobs.max(MAX_INCLUDE_GLOBS_PER_REQUEST)`, `excludeGlobs.max(MAX_EXCLUDE_GLOBS_PER_REQUEST)` |
| `GetFileChecksumsArgsSchema` | `paths.max(MAX_GENERIC_PATHS_PER_REQUEST)`, each path `.max(PATH_MAX_CHARS)` |
| `VerifyFileChecksumsArgsSchema` | `files.max(MAX_GENERIC_PATHS_PER_REQUEST)`, each `path.max(PATH_MAX_CHARS)`, each `expectedHash.max(HASH_STRING_MAX_CHARS)` |

**Architectural Rationale:**
These endpoints can safely accept larger batch sizes than regex search because their outputs are more predictable, but they still need explicit contract caps so the client sees finite limits before the handler runs.

**Exact Target:**
All schema files listed in `files_modified`.

**Implementation Contract:**
Import shared constants; do not inline numbers locally.

### IU-002: Exact Runtime Result Policy for Discovery and Metadata Handlers
**Type:** `constraint`

**Description:**
Define which handlers must truncate versus refuse and which can stay summary-only.

**Ist-Zustand:**
Handlers mostly emit unbounded text lists or batch summaries.

**Soll-Zustand:**
Use these exact runtime policies:

| Handler | Required Runtime Policy |
|------|-------|
| `handleListDirectoryEntries` | Refuse when encoded text exceeds `DISCOVERY_RESPONSE_CAP_CHARS`; do not silently truncate structured output |
| `handleSearchFiles` / `handleSearchGlob` | Preserve caller-visible results but truncate to the schema-level `maxResults` and refuse if formatted output still exceeds `DISCOVERY_RESPONSE_CAP_CHARS` |
| `handleCountLines` | Refuse when formatted output exceeds `DISCOVERY_RESPONSE_CAP_CHARS`; do not change numeric totals |
| `handleChecksumFiles` / `handleChecksumFilesVerif` / `handleGetPathMetadata` | Refuse when formatted output exceeds `METADATA_RESPONSE_CAP_CHARS`; structured outputs remain complete if within cap |

**Architectural Rationale:**
Discovery endpoints can safely support bounded truncation when a caller already expects result limiting. Metadata and checksum endpoints should refuse rather than partially omit trusted lookup surfaces.

**Exact Target:**
Handlers listed in `files_modified`.

**Implementation Contract:**
Use the shared text-budget and error-contract helpers. Add structured `truncated` flags only when the endpoint already semantically supports limited result sets.

## Actions
[INTENT: ANWEISUNG]
1. **Update the targeted schemas to use canonical caps**
   - **What:** Modify each targeted inspection schema with the exact limits listed in IU-001.
   - **Where:** All schema files in `files_modified`.
   - **How:** Import the shared constants and add `.max(...)` or `.max(...).describe(...)` constraints without renaming existing properties.
   - **Why:** The MCP SDK surfaces these constraints to callers and validates them before handlers run.
   - **Inputs / References:** shared limit matrix, current schema files.
   - **Output Contract:** Every targeted array and string surface now has an explicit finite ceiling.
   - **Verification:** Confirm no targeted schema still contains an unbounded `z.array(z.string())` or unconstrained free-text string where IU-001 requires a cap.
   - **Stop if:** Any required schema field cannot express the listed cap without breaking the existing property name or optionality.
2. **Add runtime budget enforcement to discovery handlers**
   - **What:** Update the targeted discovery and metadata handlers per IU-002.
   - **Where:** The corresponding handler files in `files_modified`.
   - **How:** Reuse the shared text-budget helpers to assert formatted output sizes and the shared error contract to refuse oversize responses.
   - **Why:** Even with schema caps, recursive discovery can still produce more formatted output than is safe.
   - **Inputs / References:** shared text-budget helper, shared error contract, existing handler formatting behavior.
   - **Output Contract:** Dynamic result fan-out is bounded consistently across the affected inspection endpoints.
   - **Verification:** Confirm each targeted handler now has one explicit shared-budget call before returning a successful large text result.
   - **Stop if:** Implementing the runtime policy would require removing existing successful structured output fields.
3. **Keep this task out of direct file reads and regex runtime logic**
   - **What:** Do not modify `read-files-with-line-numbers/*` or `search-file-contents-by-regex/*` here.
   - **Where:** These files must remain untouched in this task.
   - **How:** Defer those families to tasks `2.2`–`2.5`.
   - **Why:** They require stricter and different guardrail treatment.
   - **Inputs / References:** `orchestration.md`.
   - **Output Contract:** Only the files listed in this task are modified.
   - **Verification:** Confirm the direct read and regex files are unchanged.
   - **Stop if:** The implementation would need to alter direct-read or regex endpoint files here.
