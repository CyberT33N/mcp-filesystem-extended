# Task 2.3: Read Files Handler Preflight and Budget Enforcement
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Convert the direct file-read handler from immediate content reading to metadata-first admission control with projected response budgeting. This task implements the primary runtime guardrail for `read_files_with_line_numbers`, ensuring that requests are rejected before file content is read when the projected line-numbered response would exceed the direct-read family budget.

## Expected Result
[INTENT: KONTEXT]
- `src/domain/inspection/read-files-with-line-numbers/handler.ts` performs validated metadata collection before any content read loop begins.
- The handler rejects oversize requests using the shared error contract when the projected line-numbered response would exceed `READ_FILES_RESPONSE_CAP_CHARS`.
- Successful reads still return the same semantic output shape when within budget.

### IU-001: Exact Preflight and Projection Sequence
**Type:** `function`

**Description:**
Define the exact runtime sequence that the read-files handler must execute before reading file content.

**Ist-Zustand:**
The handler reads all requested files directly and only learns the total output size after building the full response string.

**Soll-Zustand:**
The updated handler MUST execute these exact steps, in order:
1. Call `collectValidatedFilesystemPreflightEntries("read_files_with_line_numbers", filePaths, allowedDirectories)`.
2. Call `assertExpectedFileTypes("read_files_with_line_numbers", entries, ["file"])`.
3. Compute `projectedChars = sum(entries.map((entry) => estimateLineNumberedResponseCharsFromBytes(entry.size))) + separatorOverhead`, where `separatorOverhead = (entries.length - 1) * 8 + entries.length * 64`.
4. Call `assertProjectedTextBudget("read_files_with_line_numbers", projectedChars, READ_FILES_RESPONSE_CAP_CHARS, "Projected line-numbered file read output exceeds the direct-read family cap.", "Reduce the number of files, target smaller files, or split the read into narrower batches.")`.
5. Only if the projected budget passes, enter the `fs.readFile` loop.
6. After building the final response string, call `assertActualTextBudget("read_files_with_line_numbers", actualChars, READ_FILES_RESPONSE_CAP_CHARS, "Actual line-numbered file read output exceeds the direct-read family cap.")` before returning success.

**Architectural Rationale:**
This preserves the ability to read legitimate batches while moving the primary guardrail to the point where the server still has only metadata, not already materialized response content.

**Exact Target:**
`src/domain/inspection/read-files-with-line-numbers/handler.ts`

**Implementation Contract:**
The handler MUST use the exact order above and MUST refuse rather than truncate direct file content.

**Scenario-Specific Specification:**

### IU-002: Exact Refusal Semantics
**Type:** `constraint`

**Description:**
Define how the handler must refuse oversize read requests.

**Ist-Zustand:**
The handler returns per-file inline error text for file-level failures and never issues a canonical batch-level budget refusal.

**Soll-Zustand:**
For projected or actual budget failures, the handler MUST throw a canonical shared guardrail failure with:
- `code = "metadata_preflight_rejected"` for pre-read projected failures emitted through the shared projected-budget helper
- `code = "runtime_budget_exceeded"` for post-read actual failures
- `toolName = "read_files_with_line_numbers"`
- `recommendedAction = "Reduce the number of files, target smaller files, or split the read into narrower batches."`

**Architectural Rationale:**
The refusal must tell the LLM agent how to recover without letting the endpoint silently truncate or partially return file content.

**Exact Target:**
`src/domain/inspection/read-files-with-line-numbers/handler.ts`

**Implementation Contract:**
The handler MUST NOT introduce partial-success truncation for direct file content.
