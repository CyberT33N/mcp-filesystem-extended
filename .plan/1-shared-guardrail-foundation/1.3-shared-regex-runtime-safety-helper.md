## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Canonical Regex Reject Rules
**Type:** `constraint`

**Description:**
Define the exact set of regex cases that the shared helper is allowed to reject before the search loop starts.

**Ist-Zustand:**
All valid JavaScript regex patterns are accepted, including patterns that can return zero-length matches forever.

**Soll-Zustand:**
The shared helper rejects only these exact cases:

| Reject Case | Exact Rule |
|------|-------|
| Empty pattern | Reject when `pattern.length === 0` |
| Zero-length matching pattern | Reject when the compiled regex can produce a match whose `match[0].length === 0` on any of these sentinel inputs: `""`, `"a"`, `" "`, `"\n"` |
| Invalid regex syntax | Reject when `new RegExp(pattern, flags)` throws |

The shared helper MUST NOT reject these cases by default:
- literal whitespace search such as `" "`
- `\s` or other high-frequency character classes
- broad but syntactically valid patterns such as `.` or `.+`

Those cases are controlled later by budgets, not by blacklist rules.

**Architectural Rationale:**
This keeps the reject layer limited to patterns that are structurally unsafe or semantically near-worthless for this endpoint, which minimizes false positives.

**Exact Target:**
`src/domain/shared/guardrails/regex-search-safety.ts`

**Implementation Contract:**
Reject logic must be implemented as runtime safety checks, not as a large static list of forbidden substrings.

### IU-002: Shared Regex Runtime Helper Surface
**Type:** `function`

**Description:**
Define the exact helper exports later used by the regex-search handler.

**Ist-Zustand:**
No shared helper exists; the handler owns compilation, result collection, and line shaping directly.

**Soll-Zustand:**
`src/domain/shared/guardrails/regex-search-safety.ts` exports these exact surfaces:

| Field | Value |
|------|-------|
| Function Name | `compileGuardrailedSearchRegex` |
| Signature | `(toolName: string, pattern: string, caseSensitive: boolean) => RegExp` |
| Return Type | `RegExp` |
| Inputs | tool name, raw pattern, case-sensitivity flag |
| Side Effects | none |
| Call Sites | regex-search handler |
| Error Behaviour | throws canonical `regex_runtime_rejected` failure for every reject case in IU-001 |
| Exact Placement | `src/domain/shared/guardrails/regex-search-safety.ts` |
| No-Breaking Constraint | Later handler keeps existing endpoint name and call contract |

Additional exact helper surface:
- `resetRegexLastIndex(regex: RegExp): void`
- `normalizeRegexMatchExcerpt(lineContent: string, matchText: string): string` returning an excerpt capped at `REGEX_SEARCH_EXCERPT_MAX_CHARS` and centered on the first visible occurrence when possible
- `assertRegexRuntimeBudget(toolName: string, collectedLocations: number, totalBytesScanned: number): void` using `REGEX_SEARCH_MAX_RESULTS_HARD_CAP` and `REGEX_SEARCH_MAX_CANDIDATE_BYTES`

**Architectural Rationale:**
These helpers separate pattern safety, stateful regex reset, line excerpt shaping, and runtime budget enforcement into one reusable layer.

**Exact Target:**
`src/domain/shared/guardrails/regex-search-safety.ts`

**Implementation Contract:**
`normalizeRegexMatchExcerpt()` MUST cap the returned excerpt to `240` characters and MUST preserve the matched text within the excerpt when possible.
