# Task 3.2: Comparison and mutation registration module extraction
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Create a dedicated application-layer registration module for the comparison and mutation bounded contexts.
The new module must register these tool families without re-declaring domain-owned contracts or keeping them inside the monolithic catalog root.

## Expected Result
[INTENT: KONTEXT]
- A new file [`register-comparison-and-mutation-tool-catalog.ts`](src/application/server/register-comparison-and-mutation-tool-catalog.ts) exists.
- The file registers all comparison and mutation tools currently embedded in the monolithic catalog.
- The new module imports domain handlers and schema contracts directly and keeps the application layer purely compositional.

## Preconditions
[INTENT: CONSTRAINT]
- Plan dependency `D2` is resolved, meaning Unit 2 has completed and the comparison/mutation domain handoff surface is stable.
- If [`tool-registration-presets.ts`](src/application/server/tool-registration-presets.ts) exists because task `3.3` was already executed, it must be used as the preset source.

## Context
[INTENT: KONTEXT]
- **Domain:** Application Server / Comparison and Mutation composition
- **Related Files:** `src/application/server/register-tool-catalog.ts`, `src/application/server/filesystem-server.ts`, optional future `src/application/server/tool-registration-presets.ts`, all comparison and mutation domain modules
- **Technical Context:** Extracting one bounded application registration module from a monolithic MCP catalog without changing any tool behavior.
- **Business Context:** The application layer must stop mixing all tool families in one file while still registering the same comparison and mutation capabilities.
- **Architectural Context:** Under DDD and 12-Factor, the application layer composes domain capabilities into the transport surface. Comparison and mutation registration should therefore live in a dedicated application module, not inside a monolithic root file.
- **Plan Context Dependencies:** The comparison and mutation domain handoff must already be complete (`PLAN:D2`).
- **Runtime Discovery Context:** None. The comparison and mutation tool set is already known from the monolithic catalog.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Comparison and mutation registration module boundary
**Type:** `object`

**Description:**
The new application module encapsulates all comparison and mutation tool registrations currently embedded in the monolithic catalog.

**Ist-Zustand:**
[`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) registers comparison and mutation tools inline together with inspection and server-scope tools.

**Soll-Zustand:**
[`register-comparison-and-mutation-tool-catalog.ts`](src/application/server/register-comparison-and-mutation-tool-catalog.ts) exports a dedicated registration function for the comparison and mutation tool family.

**Architectural Rationale:**
This extraction mirrors bounded contexts at the application composition level and allows the final root composition to delegate by family rather than by individual inline registration blocks.

**Exact Target:**
`src/application/server/register-comparison-and-mutation-tool-catalog.ts`

**Implementation Contract:**
The new module must register these tools and no others: `diff_files`, `diff_text_content`, `create_files`, `append_files`, `delete_paths`, `copy_paths`, `move_paths`, `replace_file_line_ranges`, `create_directories`.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `registerComparisonAndMutationToolCatalog` |
| Full Target Shape / Value | `Application registration module containing only the comparison and mutation tool registrations and the shared registration context contract` |
| Authoritative Source | Current comparison/mutation registrations in [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) plus the stabilized domain contracts |
| Determination Rule | Copy behavior, move ownership, import domain contracts directly, and do not re-declare schemas locally |
| Exact Placement | `src/application/server/register-comparison-and-mutation-tool-catalog.ts` |
| Consumers | Final composition root in task `3.4` |
| No-Breaking Constraint | Same tool names, descriptions, handlers, annotations, execution hints, and parameter semantics |

## Actions
[INTENT: ANWEISUNG]
1. **Reconfirm the comparison and mutation tool set and behavior**
   - **What:** Re-read the monolithic application catalog and the stabilized comparison/mutation domain modules.
   - **Where:** [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) plus the domain schema references above.
   - **How:** Extract the exact comparison/mutation registration blocks, including tool names, descriptions, annotations, execution hints, and parameter schema usage.
   - **Why:** The extracted module must preserve behavior exactly.
   - **Inputs / References:** All authoritative references listed above.
   - **Output Contract:** A verified comparison/mutation registration blueprint for the new module.
   - **Verification:** The blueprint covers all listed comparison and mutation tools and excludes inspection/server-scope tools.
   - **Stop if:** The domain handoff surface is not yet stable or the registration behavior differs from this task file.
2. **Create the dedicated comparison-and-mutation registration module**
   - **What:** Create [`register-comparison-and-mutation-tool-catalog.ts`](src/application/server/register-comparison-and-mutation-tool-catalog.ts).
   - **Where:** `src/application/server/`
   - **How:** Implement one exported registration function that accepts the same registration context shape used by the monolithic catalog, registers only the comparison and mutation tools, imports domain handlers and schemas directly, and imports shared presets from [`tool-registration-presets.ts`](src/application/server/tool-registration-presets.ts) if that file exists.
   - **Why:** This extracts the second bounded registration module needed before the root composition can be slimmed down.
   - **Inputs / References:** Current monolithic catalog, comparison and mutation domain modules, optional preset module.
   - **Output Contract:** The new module is behaviorally equivalent for the comparison/mutation tool family and contains no domain schema duplication.
   - **Verification:** The file registers only comparison and mutation tools and imports contracts from the domain modules.
   - **Stop if:** The preset module is missing or the extraction would require redefining domain contracts locally.
3. **Leave the root composition untouched in this task**
   - **What:** Keep [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) and [`filesystem-server.ts`](src/application/server/filesystem-server.ts) read-only.
   - **Where:** Those two application files
   - **How:** Use them as behavior references only.
   - **Why:** Final root rewiring belongs to task `3.4`.
   - **Inputs / References:** [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts), [`filesystem-server.ts`](src/application/server/filesystem-server.ts)
   - **Output Contract:** No existing application file is modified by this task.
   - **Verification:** Only the new module file is created.
   - **Stop if:** Extraction appears to require immediate root composition rewiring.
