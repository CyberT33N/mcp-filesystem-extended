# Task 2.2: Mutation handler and helper normalization
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Normalize the mutation-domain handler, helper, and schema naming so the bounded context consistently speaks the final target-state vocabulary.
This task must remove stale legacy naming such as `writeNewFiles`, `deleteFiles`, `moveFiles`, `copyFile`, and `patchFiles` remnants while preserving behavior.

## Expected Result
[INTENT: KONTEXT]
- Mutation-domain internal names, schema exports, and helper DTO names align to `create_files`, `append_files`, `copy_paths`, `move_paths`, `delete_paths`, and `replace_file_line_ranges` terminology.
- No public behavior, parameter field names, or filesystem semantics change.
- The mutation domain is ready for final handoff normalization in task `2.3` and later application-layer import rewiring in Unit 3.

**Ist-Zustand:**

**Soll-Zustand:**

**Architectural Rationale:**
The mutation bounded context should not require application-layer translation for basic terminology. Stable domain naming improves maintainability and future decomposition quality.

**Exact Target:**

### IU-002: Copy-paths helper and operation DTO normalization
**Type:** `object`

**Description:**
Normalize copy-paths operation DTO names and helper naming to match path-oriented target-state semantics.

**Ist-Zustand:**
[`helpers.ts`](src/domain/mutation/copy-paths/helpers.ts) still uses `CopyFileOperation` and `PreparedCopyFileOperation`, which are stale because the tool handles files and directories under the `copy_paths` surface.

**Soll-Zustand:**
The helper DTOs and any associated handler references use path-oriented naming such as `CopyPathsOperation` and `PreparedCopyPathsOperation` or an equivalent target-state vocabulary that clearly reflects path-level semantics.

**Architectural Rationale:**
The helper boundary should encode the same concept as the tool surface. File-only naming is misleading because the tool supports both files and directories.

**Exact Target:**
`src/domain/mutation/copy-paths/handler.ts`, `src/domain/mutation/copy-paths/helpers.ts`, `src/domain/mutation/copy-paths/schema.ts`

**Implementation Contract:**
Rename helper DTOs and consuming references only. Preserve all copy safety rules and parallel-conflict detection behavior.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `Copy-paths helper DTO names` |
| Full Target Shape / Value | `Path-oriented DTO and helper naming matching file-or-directory semantics` |
| Authoritative Source | Current tool contract and support for copying files or directories |

| Exact Placement | `src/domain/mutation/copy-paths/handler.ts`, `src/domain/mutation/copy-paths/helpers.ts`, `src/domain/mutation/copy-paths/schema.ts` |
| Consumers | Copy handler and future application registration imports |
| No-Breaking Constraint | Safety logic and field structure remain unchanged |

### IU-003: Replace-file-line-ranges helper and schema naming normalization
**Type:** `function`

**Description:**
Normalize remaining patch-oriented naming in the line-range replacement boundary to direct line-range replacement language.

**Ist-Zustand:**
[`handler.ts`](src/domain/mutation/replace-file-line-ranges/handler.ts) and [`helpers.ts`](src/domain/mutation/replace-file-line-ranges/helpers.ts) still expose legacy `patch` naming even though the public tool surface is already `replace_file_line_ranges`.

**Soll-Zustand:**
Handler-local and helper-local naming reflects line-range replacement semantics rather than unified diff patch terminology, while preserving the actual replacement behavior.

**Architectural Rationale:**
The domain language should match the public contract. Leaving patch-centric helper naming in place preserves the ambiguity the migration is designed to remove.

**Exact Target:**
`src/domain/mutation/replace-file-line-ranges/handler.ts`, `src/domain/mutation/replace-file-line-ranges/helpers.ts`, `src/domain/mutation/replace-file-line-ranges/schema.ts`

**Implementation Contract:**
Rename handler exports, helper DTOs/options names where necessary, and schema export constants to line-range replacement terminology. Do not change the replacement algorithm, line numbering rules, or dry-run behavior.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Function Name | Normalize to line-range replacement terminology per file |
| Signature | Preserve existing replacement semantics and options |
| Return Type | Existing return types remain unchanged |
| Inputs | Existing replacement arrays and dryRun semantics remain unchanged |
| Side Effects | Existing file write behavior remains unchanged |
| Call Sites | Future Unit 3 application registration imports |
| Error Behaviour | Existing invalid-line-range and file-write errors remain unchanged |
| Exact Placement | `src/domain/mutation/replace-file-line-ranges/handler.ts`, `src/domain/mutation/replace-file-line-ranges/helpers.ts`, `src/domain/mutation/replace-file-line-ranges/schema.ts` |
| No-Breaking Constraint | No line-range semantics or field names change |
