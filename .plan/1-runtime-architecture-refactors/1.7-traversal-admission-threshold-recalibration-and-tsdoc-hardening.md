[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Raise the shared traversal-admission and preview-lane values to the agreed higher target bands so large valid recursive workloads stop falling into preview-first too early, while preserving the deeper runtime safeguard as a final emergency fuse.

This task is the calibrated-values correction layer. It does **not** add continuation tokens or SQLite persistence yet. It prepares the corrected admission bands that task `1.8` will later consume.

## Expected Result
[INTENT: KONTEXT]

- Shared traversal admission thresholds are materially higher.
- The runtime capability layer no longer collapses proven local environments into the all-`D` / `UNKNOWN` startup baseline.
- The shared execution policy exposes the agreed 0-49 / 50-84 / 85-100 response-band model, the 15-second sync window, and the 60-second task threshold with explicit TSDoc rationale.
- All affected recursive endpoint families inherit the higher bands through the existing shared-policy consumption path.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Shared response-band model
**Type:** `constraint`

**Description:**
Raise the shared admission-band model so preview-first begins later and task-backed-required begins only in the final high-pressure band.

**Ist-Zustand:**
The current shared policy exposes a 50% preview-first trigger and a 15-second / 60-second time vocabulary, but the effective local admission still becomes too conservative too early because the derived budget tables and startup floor are too low.

**Soll-Zustand:**
The shared execution policy continues using the same `15s / 60s / 50%` architectural anchors, adds the explicit `85%` task-backed band field, and uses materially higher derived budgets so valid local recursive workloads stay inline longer before preview-first activates.

**Architectural Rationale:**
The architecture is correct at the band level; the implementation needs the higher calibrated values so the preview lane appears where it is actually intended to appear.

### IU-002: Higher shared preflight and deeper-safeguard ceilings
**Type:** `constant`

**Description:**
Raise the shared traversal preflight and deeper-safeguard breadth ceilings while preserving the deeper fuse role.

**Ist-Zustand:**
The shared limit registry still carries lower preflight entry/directory/time ceilings and lower deeper runtime breadth ceilings.

**Soll-Zustand:**
The shared limit registry carries the higher exact values bound in this task, while `TRAVERSAL_RUNTIME_SOFT_TIME_BUDGET_MS` remains the explicit deeper emergency fuse.

**Architectural Rationale:**
The system should fail later for broad valid local roots, but it must still fail deterministically before runtime instability or pathological breadth is reached.

### IU-003: Static discovery floor for proven local environments
**Type:** `object`

**Description:**
Stop collapsing proven local environments into the all-`D` / `UNKNOWN` fallback when startup discovery can safely establish a higher conservative floor.

**Ist-Zustand:**
`detectIoCapabilityProfile()` can resolve the all-`D` / `UNKNOWN` baseline on empty input, which keeps the effective admission bands lower than intended.

**Soll-Zustand:**
For proven local environments, the detector resolves the explicit `C/C/C + LOW` startup floor and leaves the all-`D` / `UNKNOWN` baseline only for truly unresolved environments.

**Architectural Rationale:**
The system must stay conservative, but it must not stay blindly pessimistic when the runtime can already prove a stronger local floor safely.

### IU-004: Architecture-grade TSDoc on the new values
**Type:** `requirement`

**Description:**
Document the new values and their rationale directly in the shared runtime source files.

**Ist-Zustand:**
The current TSDoc explains the existing policy, but it does not yet document the higher admission calibration and the explicit reason why those values are intentionally raised.

**Soll-Zustand:**
The shared TSDoc surfaces explain the higher preflight ceilings, the higher startup floor, the 50% preview-first boundary, the 85% task-backed boundary, and why the deeper fuse remains unchanged in role.

**Architectural Rationale:**
The next execute agent must be able to re-implement or preserve the calibrated model without silently regressing to the earlier lower values.
