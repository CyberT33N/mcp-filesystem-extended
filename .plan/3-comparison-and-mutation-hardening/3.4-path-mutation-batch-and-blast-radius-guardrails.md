# Task 3.4: Path Mutation Batch and Blast-Radius Guardrails
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Harden the path-mutation endpoints with finite batch-count and path-length caps plus handler-level blast-radius checks that bound destructive or large-scale local filesystem operations. This task protects create-directories, delete-paths, move-paths, and copy-paths without changing their endpoint names or core semantics.

[INTENT: KONTEXT]
- All path-mutation schemas apply canonical batch-count and path-length ceilings.
- All targeted handlers refuse oversize or overly broad mutation batches before performing filesystem modifications.
- Existing success summaries remain concise and semantically unchanged when requests are within the approved blast radius.

## Context
[INTENT: KONTEXT]
- **Domain:** Path-mutation batch safety and blast-radius control.
- **Related Files:** `src/domain/mutation/create-directories/*`, `delete-paths/*`, `move-paths/*`, `copy-paths/*`, shared guardrail modules.
- **Technical Context:** These endpoints do not primarily risk large response payloads; their risk is large destructive or large additive filesystem operations performed in one request.
- **Business Context:** The server must still support legitimate multi-path operations, but it must refuse oversized or overly broad mutation batches instead of executing them blindly.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Exact Schema Guardrails for Path Mutation Endpoints
**Type:** `schema`

**Description:**
Apply the exact shared path-count and path-length caps to the four path-mutation schemas.

**Ist-Zustand:**
The targeted schemas only use `.min(1)` and allow unbounded path arrays and unbounded path strings.

**Soll-Zustand:**
Apply these exact schema changes:

| Target Schema | Required Changes |
|------|-------|
| `CreateDirectoriesArgsSchema` | `paths.max(MAX_OPERATIONS_PER_PATH_MUTATION_REQUEST)`, each path `.max(PATH_MAX_CHARS)` |
| `DeletePathsArgsSchema` | `paths.max(MAX_OPERATIONS_PER_PATH_MUTATION_REQUEST)`, each path `.max(PATH_MAX_CHARS)` |
| `MovePathsArgsSchema` | `operations.max(MAX_OPERATIONS_PER_PATH_MUTATION_REQUEST)`, each `sourcePath.max(PATH_MAX_CHARS)`, each `destinationPath.max(PATH_MAX_CHARS)` |
| `CopyPathsArgsSchema` | `operations.max(MAX_OPERATIONS_PER_PATH_MUTATION_REQUEST)`, each `sourcePath.max(PATH_MAX_CHARS)`, each `destinationPath.max(PATH_MAX_CHARS)` |

**Architectural Rationale:**
This gives callers a visible finite contract and blocks obviously excessive mutation batches before handlers run.

**Exact Target:**
All schema files listed in `files_modified`.

**Implementation Contract:**
Import shared constants; do not inline numbers locally.

### IU-002: Exact Blast-Radius Runtime Policy
**Type:** `constraint`

**Description:**
Define the handler-level blast-radius and refusal behavior for path-mutation endpoints.

**Ist-Zustand:**
Handlers execute mutation loops immediately after path validation.

**Soll-Zustand:**
Apply these exact runtime rules:

| Handler | Required Runtime Policy |
|------|-------|
| `handleCreateDirectories` | Refuse when `paths.length > MAX_OPERATIONS_PER_PATH_MUTATION_REQUEST` as a defensive runtime check before the loop begins |
| `handleDeletePaths` | Same defensive runtime check before any deletion call |
| `handleMovePaths` | Same defensive runtime check before any move call |
| `handleCopyPaths` | Same defensive runtime check before any copy call plus existing overlap checks |

Additionally, `copy-paths/helpers.ts` MUST keep its current overlap detection and extend it only if necessary to preserve deterministic refusal messaging through the shared error contract.

**Architectural Rationale:**
Even though schemas already cap batch sizes, path-mutation handlers must still defend themselves as a second safety layer because these endpoints cause direct filesystem change and therefore have a higher operational blast-radius concern than their response size suggests.

**Exact Target:**
All handler/helper files listed in `files_modified`.

**Implementation Contract:**
Do not introduce caller-overridable bypass flags. Refuse the request before any filesystem mutation begins.

## Actions
[INTENT: ANWEISUNG]
1. **Update the four path-mutation schemas with the exact shared caps**
   - **What:** Apply the schema constraints from IU-001.
   - **Where:** The four schema files in `files_modified`.
   - **How:** Import shared constants and add `.max(...)` constraints to arrays and path strings.
   - **Why:** The request contract must be finite and caller-visible.
   - **Inputs / References:** shared limit matrix, current schema files.
   - **Output Contract:** All four schemas now expose finite batch-count and path-length limits without renaming any field.
   - **Verification:** Confirm no targeted schema still contains an unbounded path array or path string.
   - **Stop if:** Applying the caps would require new request fields or renamed properties.
2. **Add defensive runtime blast-radius checks before mutation loops**
   - **What:** Update all four handlers to defensively refuse oversized batches before any filesystem mutation begins.
   - **Where:** The four handler files in `files_modified`.
   - **How:** Check the relevant array length against `MAX_OPERATIONS_PER_PATH_MUTATION_REQUEST` at runtime and refuse using the shared error contract if exceeded.
   - **Why:** Direct filesystem mutations deserve a second non-bypassable layer even after schema validation.
   - **Inputs / References:** shared limit matrix, shared error contract.
   - **Output Contract:** No filesystem mutation occurs when the runtime blast-radius check fails.
   - **Verification:** Confirm the defensive check runs before the first `fs.mkdir`, `fs.rm`, `fs.rename`, `fs.copyFile`, or recursive copy call.
   - **Stop if:** The handler would need to partially execute the batch before determining refusal.
3. **Keep copy-path overlap safety aligned to the shared refusal contract**
   - **What:** Update `copy-paths/helpers.ts` only as needed so its existing overlap checks remain aligned to the shared error wording and do not drift into ad hoc messages.
   - **Where:** `src/domain/mutation/copy-paths/helpers.ts`
   - **How:** Preserve the current overlap and subtree conflict logic, but normalize any new or existing thrown message that now belongs to the shared refusal surface.
   - **Why:** Copy-paths already has richer blast-radius logic than the other handlers and must remain the canonical overlap-protection endpoint.
   - **Inputs / References:** current helper implementation, shared error contract.
   - **Output Contract:** Conflict detection remains intact and any new refusal wording stays deterministic.
   - **Verification:** Confirm no overlap-conflict rule is removed.
   - **Stop if:** Aligning the wording would require weakening the current overlap detection logic.
