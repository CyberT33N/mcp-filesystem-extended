# Task 1.9: Preview continuation delivery and response-budget hardening
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Close the residual runtime conflict in which a preview-capable inspection family can already build a valid same-endpoint continuation response and then still fail before delivery because the caller-visible `content.text` surface exceeds a late family-local text budget.

This task hardens delivery across the affected continuation-aware inspection families so valid preview-first continuation metadata actually reaches the caller, while the current same-endpoint continuation contract, current response caps, and current global response fuse remain intact.

## Expected Result
[INTENT: KONTEXT]

- `list_directory_entries`, `find_files_by_glob`, `find_paths_by_name`, `search_file_contents_by_regex`, and `search_file_contents_by_fixed_string` can all deliver valid continuation-bearing success responses without losing `continuationToken` to a later formatted-text or encoded-text budget failure.
- Continuation-bearing responses keep their authoritative preview payload in `structuredContent` and collapse `content.text` to a bounded summary/guidance surface.
- `list_directory_entries` no longer TOON-encodes the full preview-capable structured listing tree as the caller-visible text surface.
- The existing response-cap constants and the global response fuse remain unchanged.
- `count_lines` remains task-backed and keeps its no-partial-preview-total rule; it is explicitly audited so this task does not regress it into preview-family behavior.

## Preconditions
[INTENT: CONSTRAINT]

- Task `1.8` must be completed.
- The execute agent must not solve this task by raising family response caps, raising the global response fuse, deleting the fuse, or deleting the same-endpoint continuation contract.
- The execute agent must not introduce a second public continuation endpoint or require the caller to resend query-defining fields.

## Runtime Conflict Statement
[INTENT: KONTEXT]

The current runtime already materializes the same-endpoint continuation architecture from task `1.8`, but the delivery path is not fully closed.

The concrete conflict is:

1. a family-specific runtime builds a valid structured result that already contains additive `admission` and `continuation` metadata,
2. the same family then builds a large caller-visible `content.text` surface that re-materializes the preview payload,
3. a family-local text-budget guard runs after that text surface is built,
4. and the whole request can then fail before the caller receives the already-valid continuation envelope.

This means the architecture is logically present but still delivery-fragile. The residual problem is not the absence of continuation tokens; it is the late text-surface failure path that still dominates caller-visible delivery on some families.

**Type:** `constraint`
