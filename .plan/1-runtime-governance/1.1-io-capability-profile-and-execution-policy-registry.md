## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Runtime capability profile contract
**Type:** `object`

**Description:**
Create a reusable runtime capability profile contract that models source-read throughput, spool-write throughput, CPU regex suitability, confidence, and provenance.

**Ist-Zustand:**
The current codebase does not expose a shared runtime capability profile. Endpoint handlers use static caps only.

**Soll-Zustand:**
A shared profile contract exists and can be resolved conservatively before large-text execution decisions are made.

**Architectural Rationale:**
The platform needs one server-owned capability vocabulary instead of per-endpoint heuristics or repeated OS-specific logic.

**Exact Target:**
`src/domain/shared/runtime/io-capability-profile.ts`

**Implementation Contract:**
Export the exact contract and tier enums declared above. Add enterprise-grade TSDoc that explains the purpose, provenance model, and why unknown environments must stay conservative for both developers and LLM agents.

### IU-002: Runtime detector surface
**Type:** `function`

**Description:**
Create the infrastructure-side detector that derives the conservative capability profile from startup discovery, optional calibration input, and telemetry snapshots.

**Ist-Zustand:**
No shared detector exists.

**Soll-Zustand:**
One detector surface exists that later endpoint code can call without per-request probing.

**Architectural Rationale:**
The execute path needs a reusable detector, but the runtime contract must stay decoupled from endpoint handlers.

**Exact Target:**
`src/infrastructure/runtime/io-capability-detector.ts`

**Implementation Contract:**
Do not perform shell probing or heavy empirical benchmarks in request paths. The detector must accept conservative inputs, materialize provenance, and expose a stable result.

### IU-003: Search execution policy registry
**Type:** `constant`

**Description:**
Create the reusable execution policy registry that binds preview-first, sync, task, and hard-gap thresholds against the capability profile.

**Ist-Zustand:**
Existing handlers only consume endpoint-local or shared guardrail constants, without runtime-tier-aware policy resolution.

**Soll-Zustand:**
One reusable policy resolver exists for later search, count, and read consumers.

**Architectural Rationale:**
This prevents per-endpoint threshold drift and keeps the runtime routing contract deterministic.

**Exact Target:**
`src/domain/shared/search/search-execution-policy.ts`

**Implementation Contract:**
The policy registry must import existing guardrail caps where relevant and extend them with runtime-aware routing thresholds. It must not duplicate the global response fuse or existing shared caps under different names.
