# Task 1.12: Resume-session dual delivery, cursor-precision hardening, and endpoint guidance architecture
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Replace the current chunk-only continuation vocabulary with a cleaner **resume-session** architecture that:

1. keeps the server as the final admission and safety authority,
2. lets preview-capable inspection families expose **two** valid post-preview paths,
3. keeps `count_lines` intentionally distinct as a **completion-backed** family,
4. hardens family-wide cursor precision,
5. and makes server plus endpoint descriptions explicit enough that a fresh autonomous LLM agent can choose between narrowing scope, taking the next chunk, or asking the server to complete the result.

This task is not a cosmetic wording update. It is a target-state runtime-contract remodel.

## Expected Result
[INTENT: KONTEXT]

- The public session handle is renamed from `continuationToken` to `resumeToken`.
- The authoritative response object is renamed from `continuation` to `resume`.
- Preview-capable families support two resume intents: `next-chunk` and `complete-result`.
- `count_lines` remains non-preview and supports only `complete-result`.
- The public contract uses the cleaner lane name `completion-backed-required` instead of `task-backed-required`.
- Scope reduction is explicitly modeled and described as a first-class alternative on every affected endpoint family.
- Family-wide cursor/frontier advancement is hardened so resume state is committed only after actual chunk inclusion or explicit next-state persistence.
- Server and endpoint descriptions become family-specific and prompt-efficient.

## Preconditions
[INTENT: CONSTRAINT]

- Task `1.11` must be treated as implemented but no longer final target-state authority for resume semantics.
- The execute agent must preserve same-endpoint resume and server-owned hard-cap authority.
- The execute agent must **not** keep `continuationToken` / `continuation` as target-state aliases, because the user explicitly approved a clean remodel.
- The execute agent must **not** equalize `count_lines` with preview-style partial-result behavior.

## Plan Correction Addendum
[INTENT: KONTEXT]

Tasks `1.8` through `1.11` implemented a same-endpoint continuation architecture that is mechanically valid, but the current public model still has four structural problems:

1. the token name still suggests a pure next-chunk cursor instead of a server-owned resume session,
2. preview-capable families expose only one post-preview operational path (`next chunk`) instead of two (`next chunk` or `complete result`),
3. the public wording does not yet present scope reduction as a first-class, endpoint-specific alternative strongly enough for autonomous LLM agents,
4. family-local cursor advancement still carries a precision risk because frontier advancement is not yet modeled as an explicit commit-only operation.

Task `1.12` therefore supersedes the public continuation vocabulary from tasks `1.8` through `1.11` as the new target-state architecture.

## Current Problem Statement
[INTENT: KONTEXT]

### A. What the current architecture can do

The current architecture can:

- admit requests into inline, preview-first, or task-backed-style runtime lanes,
- return bounded preview payloads on preview-capable families,
- persist server-owned state in SQLite,
- and resume through the same endpoint.

### B. What the current architecture cannot do cleanly

The current architecture cannot yet express, in one clean public model:

- that the persisted token identifies a **resume session**, not merely a next chunk,
- that preview-capable families should support both **chunk inspection** and **server-owned completion**,
- that `count_lines` is not just “different by accident” but intentionally **completion-backed**,
- that scope reduction is always a valid alternative and must be surfaced per endpoint,
- and that the cursor may never advance ahead of the actually committed preview boundary.

### C. Prompt-efficiency impact

For an autonomous LLM orchestration agent, a chunk-only post-preview model is inefficient when the task clearly needs global completeness.

That inefficiency creates:

- extra MCP round-trips,
- more sequential reasoning checkpoints,
- more prompt drift between continuation calls,
- and a higher hallucination probability during long preview chains.

The architecture must therefore let the agent say:

- “show me the next bounded chunk” **or**
- “continue this server-owned session toward a complete result.”

## Target Architecture Clarification
[INTENT: KONTEXT]

### 1. One session handle, not two primary public tokens

The correct target state is **not**:

- one token for next chunk,
- and a different token for full result.

The correct target state is:

- one opaque server-owned **resume session** handle,
- and one explicit **resume intent** field.

The session handle identifies persisted state.
The intent selects how the server should continue that state.

### 2. The two post-preview paths for preview-capable families

For preview-capable families, the target state is:

- `resumeMode = next-chunk`
  - return the next bounded preview chunk,
  - return a fresh resumable session state when more remains,
  - preserve compact caller-visible text and authoritative structured payloads.

- `resumeMode = complete-result`
  - continue the same server-owned session toward a final complete result,
  - do **not** bypass hard caps,
  - do **not** promise one-call success,
  - return either the final full result, a renewed resumable completion session, or narrowing guidance.

This means “full response” is remodeled as **server-owned completion**, not as “disable all guardrails.”

### 3. The `count_lines` family remains intentionally distinct

`count_lines` is not a preview-family endpoint and should stay that way.

The correct target state for [`count_lines`](src/domain/inspection/count-lines/handler.ts:667) is:

- inline when safe,
- otherwise `completion-backed-required`,
- then `resumeMode = complete-result` only,
- no partial preview totals,
- no `next-chunk` public mode.

This does **not** make `count_lines` inconsistent.
It makes it **correctly specialized**.

### 4. Scope reduction remains a first-class alternative

The agent must be told that it may always choose to reduce the request instead of resuming.

That means the endpoint and server descriptions must explicitly explain that the agent may:

- narrow roots,
- choose deeper paths,
- tighten globs,
- tighten name filters,
- or use include-glob boundaries,

to avoid landing in preview or completion-backed lanes when that is architecturally preferable.

### 5. Cursor precision must be hardened

The family-wide cursor rule is:

- no preview-family frontier may advance solely because iteration moved forward in memory,
- it may advance only after the entry, match, or traversal boundary is actually committed into the current delivered chunk,
- or explicitly persisted as the next pending unit in the resume state.

This is the correct hardening target for the repeated frontier-precision risk already visible in current preview-family implementations.

[INTENT: KONTEXT]

- raise family response caps,
- weaken the global fuse,
- normalize `count_lines` into preview-family behavior,
- or claim that `complete-result` allows the client to force an unsafe response.

## Information Units

### IU-001: Resume-session contract remodel
**Type:** `object`

**Description:**
Replace the public continuation vocabulary with a cleaner resume-session contract.

**Ist-Zustand:**

**Soll-Zustand:**

**Architectural Rationale:**

**Type:** `workflow`

**Description:**

**Ist-Zustand:**
Preview-capable families expose only bounded preview plus chunk-style resume.

**Soll-Zustand:**
Preview-capable families expose `next-chunk` and `complete-result` as explicit resume intents.

**Architectural Rationale:**
Prompt-efficient orchestration requires a direct server-owned completion option when the agent already knows it needs the full dataset.

### IU-003: Completion-backed count-lines model
**Type:** `workflow`

**Description:**
`count_lines` remains distinct and completion-backed only.

**Ist-Zustand:**
`count_lines` uses task-backed resume semantics, but the current vocabulary is harder to reason about and not clearly aligned with the rest of the contract.

**Soll-Zustand:**
`count_lines` uses the shared resume-session vocabulary while exposing only `resumeMode = complete-result` and no preview-style chunk semantics.

**Architectural Rationale:**
The family is different because partial totals are semantically misleading, not because it was implemented later.

### IU-004: Family-wide cursor precision hardening
**Type:** `constraint`

**Description:**
Preview-family frontier state must become commit-based instead of iteration-based.

**Ist-Zustand:**
Several preview families still carry a risk that frontier advancement happens too early relative to chunk commit.

**Soll-Zustand:**
One shared frontier-commit contract guarantees exact replay-safe resume semantics.

**Architectural Rationale:**
Resume correctness depends on committed boundaries, not on local loop position.

### IU-005: Endpoint-specific guidance and server wording
**Type:** `requirement`

**Description:**
Every affected endpoint description and the server-level wording must explain scope reduction and the available resume intents precisely.

**Ist-Zustand:**
Current wording focuses on continuation semantics and structured authority, but it does not yet encode the new endpoint-specific dual-path model strongly enough.

**Soll-Zustand:**
Descriptions explain:

- how to reduce scope,
- whether `next-chunk` is available,
- whether `complete-result` is available,
- and which mode is recommended for the family.

**Architectural Rationale:**
Autonomous agents should not infer endpoint behavior from generic continuation wording.
