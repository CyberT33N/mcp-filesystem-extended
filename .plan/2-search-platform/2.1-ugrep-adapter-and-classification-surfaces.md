# Task 2.1: Introduce the shared ugrep adapter and classification surfaces
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Create the common native-search adapter and the two classification surfaces that every later large-text search path will use.

This task is where the platform stops treating search as a JavaScript string-processing concern and starts treating it as a server-owned native search capability with explicit contract boundaries.

## Expected Result
[INTENT: KONTEXT]

- The platform has one reusable `ugrep` runner and command builder.
- The platform can classify whether a file is text-eligible before search execution.
- The platform can classify whether a user pattern is literal, safe regex, or PCRE2-heavy.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Shared `ugrep` runner
**Type:** `function`

**Description:**
Create the reusable runner that executes `ugrep` and returns structured results for later endpoint consumers.

**Ist-Zustand:**
There is no shared native search adapter in the current codebase.

**Soll-Zustand:**
One `ugrep` runner exists and later endpoint handlers can consume it instead of building ad-hoc shell integrations.

**Architectural Rationale:**
The platform needs one search backend with one command-construction surface and one result interpretation surface.

**Exact Target:**
`src/infrastructure/search/ugrep-runner.ts`

**Implementation Contract:**
Return structured execution metadata, not only raw stdout. Add detailed TSDoc that explains why `ugrep` is the single search SSOT and why the runner must not bypass guardrails.

### IU-002: `ugrep` command builder
**Type:** `object`

**Description:**
Create the deterministic command-builder surface that maps classification outputs and policy settings into `ugrep` invocations.

**Ist-Zustand:**
No command-builder exists.

**Soll-Zustand:**
There is one central `ugrep` command builder used by all native-search consumers.

**Architectural Rationale:**
This avoids endpoint-local argument drift and keeps the binary contract explicit.

**Exact Target:**
`src/infrastructure/search/ugrep-command-builder.ts`

**Implementation Contract:**
The builder must encode fixed-string vs regex vs PCRE2-heavy execution intent without exposing shell-specific decisions to endpoint handlers.

### IU-003: Pattern classification
**Type:** `schema`

**Description:**
Classify search patterns into literal, automaton-safe regex, or PCRE2-heavy regex.

**Ist-Zustand:**
The current regex endpoint compiles one JavaScript regex directly from user input.

**Soll-Zustand:**
The platform has a shared classification surface that later handlers consume before choosing execution policy.

**Architectural Rationale:**
Heavy regex must be distinguished from cheap regex so the server can decide preview, sync, task, and hard-gap behavior predictably.

**Exact Target:**
`src/domain/shared/search/pattern-classifier.ts`

**Implementation Contract:**
Classification must be deterministic and documented; the task must not leave the execute agent free to invent class boundaries.

### IU-004: Text/binary classification
**Type:** `schema`

**Description:**
Create the shared file-surface classifier that determines whether a candidate file is eligible for text search and large-file counting.

**Ist-Zustand:**
The current regex handler and count-lines handler do not share a hybrid text/binary classifier.

**Soll-Zustand:**
One reusable classifier exists and later endpoints can apply the same binary-vs-text decision.

**Architectural Rationale:**
The platform should avoid a brittle exhaustive format allow-list while still blocking obvious binary or container surfaces.

**Exact Target:**
`src/domain/shared/search/text-binary-classifier.ts`

**Implementation Contract:**
The classifier must combine a small assist list, explicit binary/container deny classes, and a generic content probe. Document the rationale in TSDoc for developers and LLM agents.
