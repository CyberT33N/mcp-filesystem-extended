# Task 1.4: Global Response Fuse Integration
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Integrate one final, non-bypassable response-size fuse into the application-layer server shell so every tool result is subject to a hard global cap before it leaves the server. This task ensures that even if an endpoint-specific preflight misses a pathological case, the server still refuses an oversized result deterministically.

[INTENT: KONTEXT]
- `src/application/server/filesystem-server.ts` applies one shared final response-size check inside `executeTool()` before returning any successful tool output.
- The final fuse reuses the canonical shared budget helpers and shared guardrail error contract rather than introducing local string logic.

### IU-001: Exact Global Fuse Behaviour
**Type:** `function`

**Description:**
Define the exact behavior of the final server-shell response fuse.

**Ist-Zustand:**
`executeTool()` returns successful string outputs without any final size check beyond endpoint-local behavior.

**Soll-Zustand:**
`executeTool()` MUST perform these exact steps on successful results:
1. If the handler returned `CallToolResult` with `isError === true`, do not re-wrap or override it.
2. If the handler returned a plain string, compute the actual output length and assert it against `GLOBAL_RESPONSE_HARD_CAP_CHARS`.
3. If the handler returned `CallToolResult`, compute the total character count of all text `content` blocks and the JSON-stringified `structuredContent` when present.
4. If the total successful response exceeds `GLOBAL_RESPONSE_HARD_CAP_CHARS`, replace the successful result with a canonical `global_response_fuse_triggered` error result.

**Architectural Rationale:**
This gives the server one universal final safety floor while preserving endpoint-specific refusal semantics and all existing successful response shapes when the budget is not exceeded.

**Exact Target:**
`src/application/server/filesystem-server.ts`

**Implementation Contract:**
Do not modify tool registration or tool names here. Only extend `executeTool()` and, if necessary, add one private helper method inside `FilesystemServer` for response-size calculation.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Function Name | `executeTool` (update) |
| Signature | `(toolName: string, action: () => Promise<CallToolResult | string>) => Promise<CallToolResult>` |
| Return Type | `Promise<CallToolResult>` |
| Inputs | Existing `toolName` and `action` callback |
| Side Effects | Logging and result normalization only |
| Call Sites | All registered MCP tools |
| Error Behaviour | Converts oversize successful responses into canonical guardrail errors without masking pre-existing error results |
| Exact Placement | `src/application/server/filesystem-server.ts` |
| No-Breaking Constraint | Existing tool names, logging hooks, and success shapes stay unchanged unless the response exceeds the hard cap |
