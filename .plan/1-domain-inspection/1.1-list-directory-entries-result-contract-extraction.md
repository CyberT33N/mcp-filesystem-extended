## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Recursive listed entry result schema
**Type:** `schema`

**Description:**
The recursive listed-entry schema models each directory entry returned by the structured listing surface, including nested `children` when recursion is enabled and opt-in metadata when `includeMetadata` is true.

**Ist-Zustand:**
The canonical runtime shape exists in [`handler.ts`](src/domain/inspection/list-directory-entries/handler.ts), but the Zod output schema is redundantly declared in [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) as `LISTED_DIRECTORY_ENTRY_OUTPUT_SCHEMA`.

**Soll-Zustand:**
[`schema.ts`](src/domain/inspection/list-directory-entries/schema.ts) exports the recursive listed-entry output schema as the canonical contract surface for the inspection boundary.

**Architectural Rationale:**
The directory-entry result shape is tool-owned domain behavior. Keeping the recursive entry schema in the application layer creates a second source of truth and violates bounded-context ownership.

**Exact Target:**
`src/domain/inspection/list-directory-entries/schema.ts` receives the recursive entry output schema export.

**Implementation Contract:**
The exported schema must encode these exact fields: `name`, `path`, `type`, optional `children`, optional `size`, optional `created`, optional `modified`, optional `accessed`, optional `permissions`.
`type` remains required and is restricted to `directory | file | other`.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `ListedDirectoryEntryOutputSchema` |
| Full Target Shape / Value | `{ name: string; path: string; type: "directory" | "file" | "other"; children?: recursive[]; size?: number; created?: string; modified?: string; accessed?: string; permissions?: string }` |
| Authoritative Source | Runtime shape in [`handler.ts`](src/domain/inspection/list-directory-entries/handler.ts) plus optional metadata semantics in [`filesystem-entry-metadata.ts`](src/infrastructure/filesystem/filesystem-entry-metadata.ts) |
| Determination Rule | Mirror the handler result exactly; do not add, rename, or remove fields |
| Exact Placement | `src/domain/inspection/list-directory-entries/schema.ts` |
| Consumers | Future application registration modules and MCP output schema wiring |
| No-Breaking Constraint | The runtime result shape and field names stay unchanged |

### IU-002: Top-level list-directory-entries structured result schema
**Type:** `schema`

**Description:**
The top-level structured result schema models the `roots` array returned by the directory-entry listing surface.

**Ist-Zustand:**
The application layer redundantly defines `LIST_DIRECTORY_ENTRIES_OUTPUT_SCHEMA` and the corresponding result type inside [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts).

**Soll-Zustand:**
[`schema.ts`](src/domain/inspection/list-directory-entries/schema.ts) exports the top-level structured result schema that wraps the recursive entry schema.

**Architectural Rationale:**
The application layer should register the tool, not own the output contract. The domain schema surface must own the entire structured result contract so later registration modules can import it directly.

**Exact Target:**
`src/domain/inspection/list-directory-entries/schema.ts` receives the top-level result schema export.

**Implementation Contract:**
The exported schema must encode `roots: Array<{ requestedPath: string; entries: ListedDirectoryEntryOutput[] }>` and must remain aligned with `getListDirectoryEntriesResult()`.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `ListDirectoryEntriesStructuredResultSchema` |
| Full Target Shape / Value | `{ roots: Array<{ requestedPath: string; entries: ListedDirectoryEntryOutput[] }> }` |
| Authoritative Source | [`getListDirectoryEntriesResult()`](src/domain/inspection/list-directory-entries/handler.ts:222) and the currently duplicated application-layer schema |
| Determination Rule | Reproduce the exact result wrapper without changing nesting or naming |
| Exact Placement | `src/domain/inspection/list-directory-entries/schema.ts` |
| Consumers | Application registration output-schema wiring and autonomous agents that need the canonical contract |
| No-Breaking Constraint | `roots`, `requestedPath`, and `entries` remain unchanged |
