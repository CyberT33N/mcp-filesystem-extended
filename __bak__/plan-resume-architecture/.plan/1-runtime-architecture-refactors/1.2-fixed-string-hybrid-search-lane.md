[INTENT: KONTEXT]

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Hybrid admission for fixed-string search
**Type:** `constraint`

**Description:**
Allow `HYBRID_SEARCHABLE` surfaces on the public fixed-string endpoint.

**Ist-Zustand:**
The endpoint effectively blocks hybrid-searchable surfaces because the shared classifier rejects them and the universal command-builder binary mode suppresses matches.

**Soll-Zustand:**
Hybrid-searchable surfaces are admitted specifically for literal search.

**Architectural Rationale:**
Literal value verification in hybrid dump-like files is a first-class enterprise inspection use case.

### IU-002: Distinct backend lane for hybrid literal search
**Type:** `function`

**Description:**
Introduce a dedicated backend lane in the command builder for hybrid literal search.

**Ist-Zustand:**
The command builder uses one universal binary-files policy.

**Soll-Zustand:**
The command builder chooses a dedicated lane for hybrid literal search while preserving the normal text lane for `TEXT_CONFIDENT` surfaces.

**Architectural Rationale:**
The public endpoint contract cannot be fulfilled while the backend still suppresses binary/hybrid matches universally.

### IU-003: Sanitized hybrid excerpt shaping
**Type:** `function`

**Description:**
Ensure that hybrid literal matches are sanitized before they are returned.

**Ist-Zustand:**
The endpoint assumes normal text excerpts and would emit raw garbage if the backend were simply widened.

**Soll-Zustand:**
Hybrid outputs are shaped explicitly and remain caller-safe.

**Architectural Rationale:**
Hybrid searchability must not degrade the caller contract into raw binary noise.
