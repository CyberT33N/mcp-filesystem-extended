[INTENT: KONTEXT]

## Information Units

### IU-001: Path metadata result schema ownership
**Type:** `schema`

**Description:**
The path-metadata result schema models `entries[]` and `errors[]` for the metadata lookup tool.

**Ist-Zustand:**
[`handler.ts`](src/domain/inspection/get-path-metadata/handler.ts) defines the runtime result shape, but the canonical Zod output schema is redundantly declared in [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts).

**Soll-Zustand:**
[`schema.ts`](src/domain/inspection/get-path-metadata/schema.ts) exports the canonical path-metadata result schema.

**Architectural Rationale:**
The inspection boundary owns metadata lookup behavior and therefore must own the result contract consumed by the application registration layer.

**Exact Target:**
`src/domain/inspection/get-path-metadata/schema.ts`

**Implementation Contract:**
Model `entries[]` with `path`, `type`, `size`, `created`, `modified`, `accessed`, `permissions`; model `errors[]` with `path` and `error`.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `GetPathMetadataResultSchema` |
| Full Target Shape / Value | `{ entries: Array<{ path: string; type: "directory" | "file" | "other"; size: number; created: string; modified: string; accessed: string; permissions: string }>; errors: Array<{ path: string; error: string }> }` |
| Authoritative Source | [`getPathMetadataResult()`](src/domain/inspection/get-path-metadata/handler.ts:51) and the duplicated application-owned schema |
| Determination Rule | Mirror the handler output exactly |
| Exact Placement | `src/domain/inspection/get-path-metadata/schema.ts` |
| Consumers | Future application registration wiring |
| No-Breaking Constraint | No field additions, removals, or renames |

## Actions
[INTENT: ANWEISUNG]
1. **Confirm all three runtime result contracts**
   - **What:** Re-read the three domain handlers and compare their runtime result objects against the duplicated application-owned schemas.
   - **Where:** [`handler.ts`](src/domain/inspection/get-path-metadata/handler.ts), [`handler.ts`](src/domain/inspection/get-file-checksums/handler.ts), [`handler.ts`](src/domain/inspection/verify-file-checksums/handler.ts), [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts)
   - **How:** Build a field-by-field parity map for each tool result shape.
   - **Why:** The domain-owned exports must match the runtime outputs exactly.
   - **Inputs / References:** All authoritative references listed above.
   - **Output Contract:** Verified shape maps for metadata, checksum generation, and checksum verification results.
   - **Verification:** Every field in each runtime result is accounted for in the target schema description.
   - **Stop if:** Any handler result contains an unplanned field or differs structurally from this task file.
2. **Add canonical result schema exports to the owning domain schema files**
   - **What:** Export the three result schema constants in their respective schema files.
   - **Where:** [`schema.ts`](src/domain/inspection/get-path-metadata/schema.ts), [`schema.ts`](src/domain/inspection/get-file-checksums/schema.ts), [`schema.ts`](src/domain/inspection/verify-file-checksums/schema.ts)
   - **How:** Add Zod object schema exports with target-state naming and keep all field names identical to the runtime result shapes.
   - **Why:** This establishes the domain modules as the single source of truth for these result contracts.
   - **Inputs / References:** The three schema files, the three handlers, and the current application-owned schemas.
   - **Output Contract:** Each schema file exports its canonical result schema and remains syntactically valid.
   - **Verification:** The result schema constants exist in each owning schema file with the documented shapes.
   - **Stop if:** Execution would require creating additional contract variants or changing runtime field semantics.
3. **Keep the migration boundary clean**
   - **What:** Avoid changing the application registration file or introducing shared helper indirection in this task.
   - **Where:** All referenced files
   - **How:** Limit modifications to the three domain schema files only.
   - **Why:** The application import rewiring belongs to Unit 3; this task is contract ownership preparation only.
   - **Inputs / References:** [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts)
   - **Output Contract:** The only modified files are the three domain schema files.
   - **Verification:** No application file diff exists at the end of the task.
   - **Stop if:** The task appears to require application-layer import rewiring to be meaningful.
