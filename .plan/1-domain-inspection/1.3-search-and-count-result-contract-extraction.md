### IU-001: Name-based path search result schema ownership
**Type:** `schema`

**Description:**
The name-search result schema models `roots[]` and `totalMatches` for the `find_paths_by_name` tool.

**Ist-Zustand:**
The runtime result shape is produced by [`getFindPathsByNameResult()`](src/domain/inspection/find-paths-by-name/handler.ts:46), but the Zod output schema is redundantly declared in [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts).

**Soll-Zustand:**
[`schema.ts`](src/domain/inspection/find-paths-by-name/schema.ts) exports the canonical result schema.

**Architectural Rationale:**
The inspection boundary owns the meaning of `roots` and `totalMatches`; the application layer should only register the tool.

**Exact Target:**
`src/domain/inspection/find-paths-by-name/schema.ts`

**Implementation Contract:**
Model `roots: Array<{ root: string; matches: string[] }>` and `totalMatches: number`.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `FindPathsByNameResultSchema` |
| Full Target Shape / Value | `{ roots: Array<{ root: string; matches: string[] }>; totalMatches: number }` |
| Authoritative Source | [`getFindPathsByNameResult()`](src/domain/inspection/find-paths-by-name/handler.ts:46) |
| Determination Rule | Mirror the handler result exactly |
| Exact Placement | `src/domain/inspection/find-paths-by-name/schema.ts` |
| Consumers | Future application registration wiring |
| No-Breaking Constraint | `root`, `matches`, and `totalMatches` remain unchanged |

   - **What:** Export one canonical result schema constant in each of the four schema files.

   - **Output Contract:** Only the four domain schema files are modified.
