# Task 2.2: Refactor the regex endpoint to the shared native search lane
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Refactor the existing regex endpoint so it preserves its public semantic contract while switching its internal execution path to the shared native `ugrep` backend.

The final handler must no longer depend on full in-process `fs.readFile()` plus JavaScript regex execution for large-file search workloads.

## Expected Result
[INTENT: KONTEXT]

- The public regex endpoint name and its request surface remain intact.
- The regex handler consumes the runtime capability profile and the shared `ugrep` adapter.
- Large valid text workloads degrade into preview-first or task-ready semantics instead of blanket file-size refusal, while unsupported or unsafe workloads still refuse.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Preserve the public regex contract
**Type:** `constraint`

**Description:**
Keep the public regex endpoint name and its request semantics stable while changing the engine underneath.

**Ist-Zustand:**
The current endpoint already exposes the public tool name `search_file_contents_by_regex` and a request contract rooted in `roots`, `regex`, glob filters, `maxResults`, and `caseSensitive`.

**Soll-Zustand:**
The public endpoint name and the request surface remain stable.

**Architectural Rationale:**
The architecture discussion explicitly preserved the public regex tool surface.

**Exact Target:**
`src/domain/inspection/search-file-contents-by-regex/schema.ts` and `src/domain/inspection/search-file-contents-by-regex/handler.ts`

**Implementation Contract:**
Do not rename the endpoint or silently collapse it into the new fixed-string endpoint.

### IU-002: Replace the internal engine
**Type:** `function`

**Description:**
Move execution off the current JavaScript full-string search path and onto the shared `ugrep` adapter.

**Ist-Zustand:**
The current handler reads file content and performs JavaScript regex execution directly.

**Soll-Zustand:**
Regex execution flows through the shared native adapter and runtime policy registry.

**Architectural Rationale:**
Large-file search must no longer depend on a Node.js full-string search engine.

**Exact Target:**
`src/domain/inspection/search-file-contents-by-regex/handler.ts`

**Implementation Contract:**
Preserve structural regex safety checks while replacing the runtime search path. Add TSDoc describing why the public tool stays stable while the internal engine changes.

### IU-003: Add aggregate budgets and preview-first semantics
**Type:** `schema`

**Description:**
Extend the regex handler so it reasons about request-aggregate search costs and preview-first response behavior.

**Ist-Zustand:**
The current candidate-byte logic is effectively root-local, and text vs structured result paths do not share one fully harmonized failure model.

**Soll-Zustand:**
The endpoint applies both per-root and request-aggregate budgets and uses one harmonized partial-failure model for multi-root search.

**Architectural Rationale:**
Large-file-safe search needs predictable preview-first degradation instead of blanket size refusal for otherwise valid workloads.

**Exact Target:**
`src/domain/inspection/search-file-contents-by-regex/handler.ts`

**Implementation Contract:**
Keep the existing shared caps active while extending the runtime model to request-aggregate accounting and preview-first result shaping.
