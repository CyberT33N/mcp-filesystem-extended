### IU-001: Regex traversal hardening and deterministic refusal
**Type:** `schema`

**Description:**
Refactor regex traversal to use the shared traversal policy and shared traversal-budget helpers, while keeping the existing regex runtime-safety layer for pattern structure and match excerpts.

**Ist-Zustand:**
[`getSearchRegexPathResult()`](src/domain/inspection/search-file-contents-by-regex/handler.ts:230) traverses broad roots recursively, applies caller-only excludes, and can visit huge vendor trees before its existing byte and result budgets become relevant.

**Soll-Zustand:**
Regex traversal resolves the effective traversal scope through the shared policy, applies traversal-entry/directory/time budgets during directory descent, and uses additive schema controls for optional `.gitignore` participation and explicit re-includes.

**Architectural Rationale:**
Broad-root timeout prevention must be owned by the server guardrail model, not left to caller luck or environment timeouts.

**Exact Target:**
[`search-file-contents-by-regex/handler.ts`](src/domain/inspection/search-file-contents-by-regex/handler.ts), [`search-file-contents-by-regex/schema.ts`](src/domain/inspection/search-file-contents-by-regex/schema.ts)

**Implementation Contract:**
Keep the existing pattern compilation and excerpt logic, but move traversal admission to the shared traversal policy and shared traversal-budget surfaces.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | Regex handler and schema |
| Full Target Shape / Value | Shared traversal-policy consumption, traversal-budget checks during descent, additive `respectGitIgnore` and `includeExcludedGlobs` controls |
| Authoritative Source | Shared traversal policy and traversal runtime-budget surfaces |
| Determination Rule | Broad roots are pruned before regex content reads, and unsafe traversal breadth fails with canonical runtime-budget refusals |
| Exact Placement | `src/domain/inspection/search-file-contents-by-regex/` |
| Consumers | Regex MCP callers, server instructions, verification tests |
| No-Breaking Constraint | Preserve existing regex pattern semantics, tool name, and structured result shape |

## Actions
[INTENT: ANWEISUNG]
1. **Refactor regex traversal to consume shared policy and budgets**
   - **What:** Replace handler-local broad-root traversal admission with the shared traversal policy and shared traversal-budget assertions.
   - **Where:** [`search-file-contents-by-regex/handler.ts`](src/domain/inspection/search-file-contents-by-regex/handler.ts)
   - **How:** Apply the shared traversal policy before directory descent, count visited entries/directories, and fail through shared runtime-budget refusals before the environment timeout becomes the first stop condition.
   - **Why:** The timeout-prone root-level regex scenario is the primary runtime risk this plan must eliminate.
   - **Inputs / References:** [`traversal-scope-policy.ts`](src/domain/shared/guardrails/traversal-scope-policy.ts), [`traversal-runtime-budget.ts`](src/domain/shared/guardrails/traversal-runtime-budget.ts), [`tool-guardrail-error-contract.ts`](src/domain/shared/guardrails/tool-guardrail-error-contract.ts)
   - **Output Contract:** Regex traversal is bounded by shared policy and shared runtime budgets before content scanning expands too far.
   - **Verification:** A broad-root regex search for `package.json` or similar no longer walks `node_modules` by default and emits deterministic refusal messaging if traversal budgets are exceeded.
   - **Stop if:** The refactor would require changing regex match result shapes or replacing the existing pattern-safety helpers.
2. **Extend the regex schema additively**
   - **What:** Add explicit traversal override fields that are clearly distinct from existing `includeGlobs` file filters.
   - **Where:** [`search-file-contents-by-regex/schema.ts`](src/domain/inspection/search-file-contents-by-regex/schema.ts)
   - **How:** Keep `includeGlobs` as file-name filters, preserve `excludeGlobs`, and add optional `respectGitIgnore` / `includeExcludedGlobs` fields for traversal override semantics.
   - **Why:** Agents must be able to tell the difference between file narrowing and excluded-subtree reopening.
   - **Inputs / References:** [`search-file-contents-by-regex/schema.ts`](src/domain/inspection/search-file-contents-by-regex/schema.ts)
   - **Output Contract:** Regex callers can narrow file candidates and separately reopen excluded subtrees when needed.
   - **Verification:** The schema remains backward-compatible and the new parameter descriptions are unambiguous.
   - **Stop if:** The parameter vocabulary would collide with existing `includeGlobs` semantics.
3. **Preserve explicit excluded-root access**
   - **What:** Ensure explicit roots inside excluded trees still work.
   - **Where:** Handler traversal admission logic.
   - **How:** Use the shared explicit-root passthrough contract instead of a global “deny vendor trees” rule.
   - **Why:** The user explicitly requires targeted vendor search to remain available.
   - **Inputs / References:** [`traversal-scope-policy.ts`](src/domain/shared/guardrails/traversal-scope-policy.ts), [`package.json`](package.json)
   - **Output Contract:** Root `node_modules/@modelcontextprotocol/sdk` remains a valid explicit search scope.
   - **Verification:** The targeted SDK root used in the architectural analysis remains readable and searchable.
   - **Stop if:** The only feasible implementation path would hard-block explicit roots inside excluded trees.
