### IU-001: Path-guard logging boundary correction
**Type:** `function`

**Description:**
The path guard must import the canonical logger helper from the correct infrastructure logging boundary and must use a module identifier that reflects the current topology.

**Ist-Zustand:**
[`path-guard.ts`](src/infrastructure/filesystem/path-guard.ts) imports `createModuleLogger` from `../utils/logger.js`, even though the canonical logger lives in [`logger.ts`](src/infrastructure/logging/logger.ts). The logger child name also still uses the stale helper-era label `helpers/path`.

**Soll-Zustand:**
[`path-guard.ts`](src/infrastructure/filesystem/path-guard.ts) imports the logger from the canonical infrastructure logging boundary and uses a module label aligned to the current topology, such as `infrastructure/filesystem/path-guard` or an equivalently precise infrastructure-local label.

**Architectural Rationale:**
Import paths and module labels are part of the infrastructure boundary language. Leaving the stale helper-era path in place weakens the correctness of the final modularized topology and makes logs misleading.

**Exact Target:**
`src/infrastructure/filesystem/path-guard.ts`

**Implementation Contract:**
Adjust only the logger import path and the module label. Do not alter path-validation behavior, home expansion behavior, symlink handling, or error semantics.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Function Name | `validatePath` and `validatePathForCreation` remain unchanged |
| Signature | Existing signatures remain unchanged |
| Return Type | Existing return types remain unchanged |
| Inputs | Existing path and allowed-directory parameters remain unchanged |
| Side Effects | Logging behavior remains present but with corrected boundary identity |
| Call Sites | All current infrastructure and application consumers remain unchanged |
| Error Behaviour | Existing access-denied and missing-parent semantics remain unchanged |
| Exact Placement | `src/infrastructure/filesystem/path-guard.ts` |
| No-Breaking Constraint | No path-validation behavior change |

### IU-002: Logger boundary stability constraint
**Type:** `constraint`

**Description:**
The infrastructure logger remains the single canonical logging surface and must only be changed if a minimal boundary-stability clarification is strictly required.

**Ist-Zustand:**
[`logger.ts`](src/infrastructure/logging/logger.ts) already provides the shared `createModuleLogger()` and `initializeLogger()` entrypoints.

**Soll-Zustand:**
[`logger.ts`](src/infrastructure/logging/logger.ts) remains the canonical logger surface. Modify it only if the corrected path-guard import or boundary naming requires a minimal non-behavioral stabilization.

**Architectural Rationale:**
The closeout task should not create unnecessary churn in the logging boundary. The logger is already canonical; the main stale artifact is the incorrect consumer-side reference in `path-guard.ts`.

**Exact Target:**
`src/infrastructure/logging/logger.ts`

**Implementation Contract:**
Prefer zero changes to `logger.ts`. If a change is necessary, it must be limited to boundary-stability clarification without changing runtime logging semantics.
