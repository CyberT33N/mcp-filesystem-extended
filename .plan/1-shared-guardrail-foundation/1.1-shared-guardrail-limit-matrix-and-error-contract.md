### IU-001: Canonical Property-Class Limit Matrix
**Type:** `constraint`

**Description:**
Define one reusable property-class matrix so schemas can import stable maxima by semantic field kind instead of inventing endpoint-local numbers.

**Ist-Zustand:**
Current schemas use only local `min(1)` rules or unconstrained strings and arrays.

**Soll-Zustand:**
The new limit matrix exports these exact property-class ceilings:

| Property Class | Exact Limit | Meaning |
|------|------:|---------|
| `IDENTIFIER_MAX_CHARS` | `128` | Stable IDs, task names, short symbolic identifiers |
| `LABEL_MAX_CHARS` | `256` | Human-readable labels, titles, and short descriptors |
| `PATH_MAX_CHARS` | `4096` | Filesystem path strings and path-like properties |
| `GLOB_PATTERN_MAX_CHARS` | `1024` | One glob pattern string |
| `REGEX_PATTERN_MAX_CHARS` | `2048` | One regex pattern string |
| `HASH_STRING_MAX_CHARS` | `256` | Expected/actual checksum text |
| `SHORT_TEXT_MAX_CHARS` | `2048` | Small explanatory or metadata text surfaces |
| `RAW_CONTENT_MAX_CHARS` | `100000` | One content-bearing free-text field |
| `REPLACEMENT_TEXT_MAX_CHARS` | `50000` | One replacement-text payload within line-range replacement |
| `MAX_INCLUDE_GLOBS_PER_REQUEST` | `32` | Upper bound for include glob arrays |
| `MAX_EXCLUDE_GLOBS_PER_REQUEST` | `64` | Upper bound for exclude glob arrays |
| `MAX_GENERIC_PATHS_PER_REQUEST` | `512` | High anti-abuse ceiling for generic path arrays |
| `MAX_DISCOVERY_ROOTS_PER_REQUEST` | `128` | Ceiling for non-regex discovery roots |
| `MAX_REGEX_ROOTS_PER_REQUEST` | `64` | Lower ceiling for regex-search roots |
| `MAX_OPERATIONS_PER_PATH_MUTATION_REQUEST` | `200` | Ceiling for directory creation, delete, move, and copy operation batches |
| `MAX_CONTENT_FILES_PER_REQUEST` | `50` | Ceiling for content-bearing mutation file arrays |
| `MAX_COMPARISON_PAIRS_PER_REQUEST` | `25` | Ceiling for file-based diff pairs |
| `MAX_RAW_TEXT_DIFF_PAIRS_PER_REQUEST` | `10` | Lower ceiling for in-memory diff pairs |
| `MAX_REPLACEMENTS_PER_FILE` | `25` | Ceiling for line-range replacement entries per file |
| `MAX_TOTAL_RAW_TEXT_REQUEST_CHARS` | `200000` | Cumulative free-text payload budget across one request |

**Architectural Rationale:**
This matrix keeps false positives low by giving high but finite static ceilings to path and batch surfaces, while using much stricter caps only for raw free-text and regex surfaces that are more likely to explode response size.

**Exact Target:**
`src/domain/shared/guardrails/tool-guardrail-limits.ts`

**Implementation Contract:**
Export these constants individually and also group them into a frozen `TOOL_GUARDRAIL_LIMITS` object so later tasks can import one named contract without duplicating literals.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `TOOL_GUARDRAIL_LIMITS` |
| Full Target Shape / Value | Property-class constants and one frozen aggregate object containing the exact limits listed above |
| Authoritative Source | This task file |
| Determination Rule | The exported values MUST match the table above exactly |
| Exact Placement | `src/domain/shared/guardrails/tool-guardrail-limits.ts` |
| Consumers | All endpoint schemas, runtime helpers, and the server-shell fuse |
| No-Breaking Constraint | Existing field names and endpoint names remain unchanged; only caps and new shared imports are introduced |

### IU-002: Endpoint-Family Runtime Budget Matrix
**Type:** `constraint`

**Description:**
Define the exact runtime budget ceilings that later handler tasks must enforce after static schema validation.

**Ist-Zustand:**
Handlers currently read and serialize content without canonical response-size or preflight-budget ceilings.

**Soll-Zustand:**
The limit matrix also exports these exact endpoint-family budgets:

| Endpoint Family Budget | Exact Value | Meaning |
|------|------:|---------|
| `GLOBAL_RESPONSE_HARD_CAP_CHARS` | `200000` | Final non-bypassable cap enforced in the server shell |
| `READ_FILES_RESPONSE_CAP_CHARS` | `180000` | Family cap for line-numbered direct file reads |
| `REGEX_SEARCH_RESPONSE_CAP_CHARS` | `60000` | Family cap for regex search text and structured output |
| `REGEX_SEARCH_MAX_RESULTS_HARD_CAP` | `200` | Highest allowed caller `maxResults` for regex search |
| `REGEX_SEARCH_MAX_CANDIDATE_BYTES` | `8388608` | Maximum total candidate file bytes (8 MiB) scanned in one regex request before refusal |
| `REGEX_SEARCH_EXCERPT_MAX_CHARS` | `240` | Maximum line excerpt length per reported match |
| `DISCOVERY_RESPONSE_CAP_CHARS` | `80000` | Cap for listing, find, and count family responses |
| `METADATA_RESPONSE_CAP_CHARS` | `60000` | Cap for metadata and checksum family responses |
| `FILE_DIFF_RESPONSE_CAP_CHARS` | `120000` | Cap for file-based unified diffs |
| `TEXT_DIFF_RESPONSE_CAP_CHARS` | `80000` | Lower cap for in-memory text diffs |
| `PATH_MUTATION_SUMMARY_CAP_CHARS` | `40000` | Cap for path-mutation batch summaries |
| `CONTENT_MUTATION_TOTAL_INPUT_CHARS` | `200000` | Cumulative create/append free-text request budget |
| `LINE_REPLACEMENT_TOTAL_INPUT_CHARS` | `200000` | Cumulative replacement-text request budget |

**Architectural Rationale:**
The family budgets intentionally treat regex search as the strictest read-oriented endpoint because its result space is the least predictable. Direct file reads receive a higher family cap because metadata preflight can predict their size more accurately.

**Exact Target:**
`src/domain/shared/guardrails/tool-guardrail-limits.ts`

**Implementation Contract:**
Export one `ENDPOINT_FAMILY_GUARDRAIL_LIMITS` object whose values exactly match the table above. Later tasks must import the family-specific limit rather than re-stating numbers.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `ENDPOINT_FAMILY_GUARDRAIL_LIMITS` |
| Full Target Shape / Value | Frozen object with the exact family-specific runtime ceilings from the table above |
| Authoritative Source | This task file |
| Determination Rule | Values MUST match the table exactly and MUST all remain `<= GLOBAL_RESPONSE_HARD_CAP_CHARS` where applicable |
| Exact Placement | `src/domain/shared/guardrails/tool-guardrail-limits.ts` |
| Consumers | Inspection handlers, comparison handlers, mutation handlers, and the global response fuse |
| No-Breaking Constraint | Budgets add server-side safety without renaming any existing endpoint or response field |

### IU-003: Canonical Guardrail Error Contract
**Type:** `object`

**Description:**
Create one shared refusal-message contract so all later rejections use deterministic, user-facing text with stable reason codes.

**Ist-Zustand:**
Handlers currently throw ad hoc errors or stringify raw exceptions.

**Soll-Zustand:**
`src/domain/shared/guardrails/tool-guardrail-error-contract.ts` exports:
- a `ToolGuardrailFailureCode` string union,
- a `ToolGuardrailFailure` object surface,
- and builders for `schema_limit_exceeded`, `metadata_preflight_rejected`, `regex_runtime_rejected`, `runtime_budget_exceeded`, and `global_response_fuse_triggered`.

The shared failure object MUST contain these exact fields:

| Field | Value |
|------|-------|
| `code` | One of the canonical failure codes |
| `toolName` | Exact MCP tool name |
| `summary` | One concise English sentence |
| `details` | Ordered array of concise English detail lines |
| `recommendedAction` | One concise English instruction to reduce scope or tighten the request |

**Architectural Rationale:**
One canonical guardrail error surface prevents wording drift across endpoint families and makes the refusal semantics compatible with LLM agents that need predictable remediation guidance.

**Exact Target:**
`src/domain/shared/guardrails/tool-guardrail-error-contract.ts`

**Implementation Contract:**
The module must export both typed builders and one `formatToolGuardrailFailureAsText()` helper that renders the failure deterministically for endpoints that still return plain text.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `ToolGuardrailFailureCode`, `ToolGuardrailFailure`, `createToolGuardrailFailure()`, `formatToolGuardrailFailureAsText()` |
| Full Target Shape / Value | Typed contract plus builders for the five refusal categories named above |
| Authoritative Source | This task file |
| Determination Rule | The builder outputs MUST keep `summary`, `details`, and `recommendedAction` in English and deterministic across callers |
| Exact Placement | `src/domain/shared/guardrails/tool-guardrail-error-contract.ts` |
| Consumers | All later schema/handler/global-fuse tasks |
| No-Breaking Constraint | Existing successful response shapes remain untouched; only refusal handling is normalized |
