# Task 3.2: Content-Bearing Mutation Schema and Handler Guardrails
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Harden the `create_files` and `append_files` endpoints with finite schema caps and cumulative raw-text budget enforcement so callers cannot push arbitrarily large content payloads into the server. This task protects the content-bearing mutation surfaces while preserving the existing endpoint names and successful summary semantics.

## Expected Result
[INTENT: KONTEXT]
- Both schemas apply shared per-file count, path-length, and content-length caps.
- Both handlers refuse oversize cumulative raw-text requests using the shared error contract before writing any file content.

## Preconditions
[INTENT: CONSTRAINT]
- Task `1.1` is DONE and exports the shared limits.
- Task `1.2` is DONE and exports the shared budget helpers.

## Context
[INTENT: KONTEXT]
- **Domain:** Content-bearing mutation request hardening.
- **Related Files:** `src/domain/mutation/create-files/*`, `src/domain/mutation/append-files/*`, shared guardrail modules.
- **Technical Context:** These endpoints accept caller-supplied file content directly and can therefore be abused through very large text payloads even when the number of files is small.
- **Business Context:** The server must still support meaningful file-creation and append workflows, but it must reject oversize payloads before any write occurs.
- **Architectural Context:** Schema caps handle per-item size and batch count; handler logic enforces cumulative request budgets and deterministic refusal messaging.
- **Plan Context Dependencies:** None.
- **Runtime Discovery Context:** None.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Exact Schema Guardrails for Content-Bearing Mutation Endpoints
**Type:** `schema`

**Description:**
Apply the exact shared caps to the create-files and append-files request surfaces.

**Ist-Zustand:**
Both schemas allow unbounded file arrays and unbounded content strings.

**Soll-Zustand:**
Apply these exact schema changes:

| Target Schema | Required Changes |
|------|-------|
| `CreateFilesArgsSchema` | `files.max(MAX_CONTENT_FILES_PER_REQUEST)`, each `path.max(PATH_MAX_CHARS)`, each `content.max(RAW_CONTENT_MAX_CHARS)` |
| `AppendFilesArgsSchema` | `files.max(MAX_CONTENT_FILES_PER_REQUEST)`, each `path.max(PATH_MAX_CHARS)`, each `content.max(RAW_CONTENT_MAX_CHARS)` |

**Architectural Rationale:**
The schema must reject obviously oversized per-item payloads and excessive file counts before the handlers begin any write work.

**Exact Target:**
`src/domain/mutation/create-files/schema.ts`, `src/domain/mutation/append-files/schema.ts`

**Implementation Contract:**
Import shared constants; do not inline local numbers.

### IU-002: Exact Handler Budget Policy for Content-Bearing Mutations
**Type:** `constraint`

**Description:**
Define the cumulative-input budget enforcement the handlers must perform before writing files.

**Ist-Zustand:**
Handlers iterate through files and write/append content directly without a cumulative request budget.

**Soll-Zustand:**
Both handlers MUST perform these exact steps before the write loop:
1. Sum all `content.length` values across the request.
2. Refuse when the total exceeds `CONTENT_MUTATION_TOTAL_INPUT_CHARS` using the shared error contract with `code = "runtime_budget_exceeded"`.
3. Only after the cumulative budget passes, continue with existing path-validation and write operations.

Optionally, after constructing the final human-readable summary text, enforce `PATH_MUTATION_SUMMARY_CAP_CHARS` if the summary itself risks growing too large.

**Architectural Rationale:**
This prevents a request with many individually valid content blocks from bypassing safety through cumulative size.

**Exact Target:**
`src/domain/mutation/create-files/handler.ts`, `src/domain/mutation/append-files/handler.ts`

**Implementation Contract:**
No write operation may start before the cumulative budget passes.

## Actions
[INTENT: ANWEISUNG]
1. **Update both content-bearing mutation schemas**
   - **What:** Apply the exact schema caps from IU-001.
   - **Where:** `src/domain/mutation/create-files/schema.ts`, `src/domain/mutation/append-files/schema.ts`
   - **How:** Import the shared constants and add `.max(...)` to file arrays, paths, and content strings.
   - **Why:** Per-item and batch caps must be enforced before handlers run.
   - **Inputs / References:** shared limit matrix, current schemas.
   - **Output Contract:** Both schemas now have finite request surfaces with unchanged property names.
   - **Verification:** Confirm no content-bearing string or file array remains unbounded.
   - **Stop if:** Implementing the caps would require renaming any request field.
2. **Add cumulative-input enforcement to create-files**
   - **What:** Update `handleCreateFiles()` to refuse oversize cumulative raw-text requests before writing files.
   - **Where:** `src/domain/mutation/create-files/handler.ts`
   - **How:** Sum all `content` lengths, compare to `CONTENT_MUTATION_TOTAL_INPUT_CHARS`, and refuse with the shared error contract if exceeded.
   - **Why:** Prevent oversize file-creation payloads before the write loop begins.
   - **Inputs / References:** shared limits, shared error contract.
   - **Output Contract:** No file write occurs when the cumulative budget fails.
   - **Verification:** Confirm the cumulative budget is checked before any `fs.writeFile` call.
   - **Stop if:** The handler would need a caller-overridable bypass to remain usable.
3. **Add cumulative-input enforcement to append-files**
   - **What:** Update `handleAppendFiles()` with the same cumulative raw-text budget enforcement.
   - **Where:** `src/domain/mutation/append-files/handler.ts`
   - **How:** Sum all `content` lengths before the append loop and refuse with the shared error contract if the total exceeds `CONTENT_MUTATION_TOTAL_INPUT_CHARS`.
   - **Why:** Prevent oversize append batches before the first append occurs.
   - **Inputs / References:** shared limits, shared error contract.
   - **Output Contract:** No append operation occurs when the cumulative budget fails.
   - **Verification:** Confirm the cumulative budget is checked before any `fs.appendFile` call.
   - **Stop if:** The handler would need to silently truncate appended content to stay safe.
