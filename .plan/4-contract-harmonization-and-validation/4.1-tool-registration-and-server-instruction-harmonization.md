## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Exact Tool-Description Harmonization Rules
**Type:** `constraint`

**Description:**
Define the exact caller-visible messaging rules for tool descriptions.

**Ist-Zustand:**
Descriptions describe functional purpose but do not mention the new guardrail layers or hard caps.

**Soll-Zustand:**
Update descriptions by family with these exact rules:
- Direct file-read tools must state that oversized projected reads are refused and that callers should reduce file count or scope.
- Regex-search tools must state that structured unsafe patterns and oversized search scopes are refused and that callers should narrow roots, globs, or `maxResults`.
- Discovery and metadata tools may state that results remain bounded and may be refused when response size would exceed server safety caps.
- Content-bearing mutation tools must state that per-item and cumulative content budgets exist.
- Path-mutation tools must state that batch-size and blast-radius limits exist and cannot be bypassed.

The descriptions MUST NOT expose exact numeric values that are likely to drift unless they are guaranteed to remain synchronized through imports/constants at generation time. Prefer qualitative but precise wording unless the registration code already derives numbers from shared constants.

**Architectural Rationale:**
The visible contract must guide callers without creating future drift between prose and implementation.

### IU-002: Exact Server-Level Instruction Rules
**Type:** `constraint`

**Description:**
Define the exact new server-level instructions and description constraints.

**Ist-Zustand:**
`SERVER_INSTRUCTIONS` and `SERVER_DESCRIPTION` describe general capabilities but not the new safety posture.

**Soll-Zustand:**
`SERVER_INSTRUCTIONS` MUST explicitly add these message themes:
- hard request and response safety caps exist server-side
- callers may narrow scope but cannot disable hard caps
- metadata-first admission control applies to file-read-style surfaces
- regex content search may refuse structurally unsafe or oversized searches

`SERVER_DESCRIPTION` MUST mention that the server includes explicit safety caps for multi-path operations, search endpoints, and line-based reading.

**Architectural Rationale:**
These server-level texts create a consistent top-level contract without duplicating every endpoint-specific rule.
