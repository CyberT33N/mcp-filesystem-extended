# Task 1.10: Continuation response contract and consumer alignment
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Explain and correct the residual architectural misalignment in which a continuation-capable inspection response can still appear to the caller as if no usable continuation token exists, even though the same-endpoint continuation architecture has already been implemented server-side.

This task must make the fault model explicit for a fresh autonomous LLM agent:

1. what is architecturally correct,
2. what is currently wrong,
3. which families are actually affected,
4. and what still must be corrected so a caller in a new context window has the full continuation truth.

## Expected Result
[INTENT: KONTEXT]

- The task explains that the target architecture is **not** “put the token somewhere in free text,” but “return authoritative `admission` and `continuation` metadata in `structuredContent`, keep `content.text` compact, and resume on the same endpoint through `continuationToken` only.”
- The task explains that the currently observed symptom can arise from **two** contract violations:
  1. preview-family runtime surfaces that emit resume guidance even when the response is not actually resumable anymore,
  2. downstream consumers that treat `content.text` as canonical and therefore ignore or discard the authoritative structured continuation metadata.
- The task binds the direct-fix family set and the task-backed `count_lines` audit boundary precisely.
- The task explains exactly what still must be fixed so a fresh autonomous agent can continue without rediscovering the architecture from scratch.

## Preconditions
[INTENT: CONSTRAINT]

- Task `1.9` must be completed.
- The execute agent must preserve the same-endpoint continuation architecture from task `1.8` and the delivery-hardening architecture from task `1.9`.
- The execute agent must not reinterpret the problem as “just print the token into free text” while leaving the authoritative structured response contract or downstream consumer authority unresolved.

## Architectural Fault Statement
[INTENT: KONTEXT]

The residual problem is **not** simply “the server forgot to create a continuation token.”

The real architectural misalignment is:

1. The accepted target architecture deliberately makes `structuredContent` authoritative for `admission` and `continuation` metadata.
2. The runtime text surface on preview-capable families is intentionally compact and may not mirror the full structured payload.
3. Some preview-family formatter/handler surfaces can still emit caller-visible resume guidance by checking only `admission.outcome = preview-first` instead of checking whether the response is actually resumable.
4. A downstream endpoint or adapter that consumes only `content.text` can therefore make a valid continuation-bearing server response look as if no usable token was returned.

This means the observed symptom “the response tells me to use continuation, but no token is visible” can come from **either**:

- a **server-side preview-family response-truth bug**, or
- a **downstream consumer contract violation** that drops authoritative structured continuation metadata.
