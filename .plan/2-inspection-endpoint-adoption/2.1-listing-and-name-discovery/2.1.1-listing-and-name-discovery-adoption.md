## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Listing adoption and explicit recursive opt-in
**Type:** `schema`

**Description:**
Update the listing handler and schema so broad-root listing uses the shared traversal policy and recursive traversal becomes explicit instead of default.

**Ist-Zustand:**
[`recursive`](src/domain/inspection/list-directory-entries/schema.ts:53) defaults to `true`, and the listing handler normalizes excludes locally via [`shouldExcludePath()`](src/domain/inspection/list-directory-entries/handler.ts:65).

**Soll-Zustand:**
Listing defaults to `recursive=false`, consumes the shared traversal policy, and exposes additive caller controls for explicit re-includes and optional `.gitignore` enrichment.

**Architectural Rationale:**
Listing the workspace root should not explode into vendor trees or deep nested content unless the caller explicitly asks for it.

**Exact Target:**
[`list-directory-entries/handler.ts`](src/domain/inspection/list-directory-entries/handler.ts), [`list-directory-entries/schema.ts`](src/domain/inspection/list-directory-entries/schema.ts)

**Implementation Contract:**
Replace local exclusion logic with the shared traversal policy, flip the recursive default to `false`, and add additive caller controls rather than renaming existing fields.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | Listing handler and schema |
| Full Target Shape / Value | Shared traversal-policy consumption, `recursive=false` default, additive `respectGitIgnore` and `includeExcludedGlobs` controls |
| Authoritative Source | Shared traversal policy from task `1.2` |
| Determination Rule | Broad roots stay safe by default; recursive depth and excluded-branch reopening become explicit caller choices |
| Exact Placement | `src/domain/inspection/list-directory-entries/` |
| Consumers | MCP callers, server instructions, registration descriptions |
| No-Breaking Constraint | Preserve tool name, output shape, and existing explicit root access |

### IU-002: Name-discovery adoption
**Type:** `schema`

**Description:**
Update the name-search helper and schema so name-based traversal shares the same default exclusions and additive override semantics as listing.

**Ist-Zustand:**
[`searchFiles()`](src/domain/inspection/find-paths-by-name/helpers.ts:34) owns its own local exclusion normalization and broad-root traversal can enter vendor/cache trees by default.

**Soll-Zustand:**
Name-discovery consumes the shared traversal policy and exposes the same additive traversal override contract as the other traversal-based endpoints.

**Architectural Rationale:**
An agent should not have to memorize one exclusion model for listing and a different one for name discovery.

**Exact Target:**
[`find-paths-by-name/helpers.ts`](src/domain/inspection/find-paths-by-name/helpers.ts), [`find-paths-by-name/schema.ts`](src/domain/inspection/find-paths-by-name/schema.ts)

**Implementation Contract:**
Remove endpoint-local exclusion heuristics, route traversal decisions through the shared policy, and keep the existing `nameContains` contract unchanged.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | Name-search helper and schema |
| Full Target Shape / Value | Shared traversal-policy consumption plus additive override fields |
| Authoritative Source | Shared traversal policy from task `1.2` |
| Determination Rule | Name-based traversal must mirror the same broad-root safety model as listing |
| Exact Placement | `src/domain/inspection/find-paths-by-name/` |
| Consumers | Name-based MCP callers and downstream docs |
| No-Breaking Constraint | Preserve the existing `nameContains` surface and result shape |

## Actions
[INTENT: ANWEISUNG]
1. **Adopt the shared traversal policy in listing**
   - **What:** Refactor the listing handler to consume the shared traversal policy and update the listing schema defaults.
   - **Where:** [`list-directory-entries/handler.ts`](src/domain/inspection/list-directory-entries/handler.ts), [`list-directory-entries/schema.ts`](src/domain/inspection/list-directory-entries/schema.ts)
   - **How:** Replace [`shouldExcludePath()`](src/domain/inspection/list-directory-entries/handler.ts:65) style logic with shared policy resolution, default [`recursive`](src/domain/inspection/list-directory-entries/schema.ts:53) to `false`, and add additive `respectGitIgnore` / `includeExcludedGlobs` fields.
   - **Why:** Root-level listing should be safe and predictable by default.
   - **Inputs / References:** [`traversal-scope-policy.ts`](src/domain/shared/guardrails/traversal-scope-policy.ts)
   - **Output Contract:** Listing uses the shared policy and only traverses deeply when callers ask for it.
   - **Verification:** A broad-root listing excludes vendor/cache trees by default while an explicit excluded root still lists normally.
   - **Stop if:** The implementation would change the tool name or structured result shape.
2. **Adopt the shared traversal policy in name discovery**
   - **What:** Refactor the name-search helper and schema to consume the shared traversal policy.
   - **Where:** [`find-paths-by-name/helpers.ts`](src/domain/inspection/find-paths-by-name/helpers.ts), [`find-paths-by-name/schema.ts`](src/domain/inspection/find-paths-by-name/schema.ts)
   - **How:** Remove helper-local exclusion normalization, route traversal through the shared policy, and add the same additive override fields used by listing.
   - **Why:** Name discovery must mirror listing semantics for broad-root safety and explicit opt-in access.
   - **Inputs / References:** [`traversal-scope-policy.ts`](src/domain/shared/guardrails/traversal-scope-policy.ts)
   - **Output Contract:** Name search shares the same default excludes and additive override semantics as listing.
   - **Verification:** A broad-root name search skips vendor/cache trees by default and a targeted excluded root still returns matches.
   - **Stop if:** The implementation would require a different override vocabulary than the other traversal-based endpoints.
3. **Keep the caller contract additive**
   - **What:** Ensure the schema changes remain additive and agent-friendly.
   - **Where:** [`list-directory-entries/schema.ts`](src/domain/inspection/list-directory-entries/schema.ts), [`find-paths-by-name/schema.ts`](src/domain/inspection/find-paths-by-name/schema.ts)
   - **How:** Keep `roots`, `excludeGlobs`, and result structures intact; add new optional traversal controls without renaming existing fields.
   - **Why:** The user asked for the recommended hardening to be implemented without rewriting the public surface into a different tool family.
   - **Inputs / References:** [`register-inspection-tool-catalog.ts`](src/application/server/register-inspection-tool-catalog.ts)
   - **Output Contract:** Downstream registration and docs can describe the new semantics without a tool rename or a schema reset.
   - **Verification:** Existing callers can still send the old fields; new behavior is driven by defaults plus additive fields.
   - **Stop if:** A field rename or result-shape change becomes necessary.
