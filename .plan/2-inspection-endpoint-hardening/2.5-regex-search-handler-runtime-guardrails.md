### IU-001: Exact Runtime Search Pipeline
**Type:** `function`

**Description:**
Define the exact runtime pipeline the handler must follow.

**Ist-Zustand:**
The handler validates the root path, compiles the regex directly, scans content, and stores full trimmed line text for each location.

**Soll-Zustand:**
The updated handler MUST execute these exact stages, in order:
1. Validate every root with the shared preflight helper and assert roots are directories.
2. Compile the regex exclusively through `compileGuardrailedSearchRegex("search_file_contents_by_regex", pattern, caseSensitive)`.
3. Traverse candidate files as today, but maintain `totalBytesScanned` as the sum of file sizes before each file content read.
4. Before reading the next candidate file, refuse if `totalBytesScanned + nextFileSize > REGEX_SEARCH_MAX_CANDIDATE_BYTES` using canonical shared refusal logic.
5. For every reported location, shape the returned line text through `normalizeRegexMatchExcerpt()` instead of using the full trimmed line.
6. After each collected location, call `assertRegexRuntimeBudget("search_file_contents_by_regex", collectedLocations, totalBytesScanned)`.
7. Preserve the existing `maxResults` truncation semantics, but never let the caller exceed the schema-level hard cap.
8. Before returning the formatted text output, call `assertActualTextBudget("search_file_contents_by_regex", actualChars, REGEX_SEARCH_RESPONSE_CAP_CHARS, "Regex search response exceeds the regex-search family cap.")`.

**Architectural Rationale:**
This keeps the endpoint expressive while stopping both structural hazards and dynamic result explosions at the correct runtime layers.

**Exact Target:**
`src/domain/inspection/search-file-contents-by-regex/handler.ts`

**Implementation Contract:**
The handler MUST reuse the shared helpers and MUST NOT re-introduce endpoint-local reject heuristics for regex contents.

**Scenario-Specific Specification:**

### IU-002: Exact Low-False-Positive Policy
**Type:** `constraint`

**Description:**
Define the precise policy choices that preserve legitimate searches while still protecting the endpoint.

**Ist-Zustand:**
The endpoint has no explicit distinction between structural hazards and high-frequency but legitimate searches.

**Soll-Zustand:**
The handler MUST preserve these cases as valid:
- literal space pattern `" "`
- `\s`
- broad literal/character-class patterns that are syntactically valid

The handler MUST rely on budgets, scopes, and result caps for these cases rather than rejecting them semantically.

The handler MUST reject only the structural cases delegated by the shared helper:
- empty pattern
- invalid regex syntax
- zero-length matching pattern

**Architectural Rationale:**
This is the explicit architectural commitment that keeps false positives low and avoids turning the endpoint into a brittle semantic blacklist.

**Exact Target:**
`src/domain/inspection/search-file-contents-by-regex/handler.ts`

**Implementation Contract:**
No new broad blacklist branches may be added locally in the handler.

## Actions
[INTENT: ANWEISUNG]
1. **Rework regex compilation and root admission control**
   - **What:** Update the handler to validate roots through the shared preflight helper and compile the regex exclusively through the shared regex runtime helper.
   - **Where:** `src/domain/inspection/search-file-contents-by-regex/handler.ts`
   - **How:** Replace direct `new RegExp(...)` construction with `compileGuardrailedSearchRegex(...)`; assert directory root types before traversal.
   - **Why:** Structural reject logic and root validation must be centralized and deterministic.
   - **Inputs / References:** shared preflight helper, shared regex helper, shared limits, shared error contract.
   - **Output Contract:** Structural hazards are rejected canonically before traversal starts.
   - **Verification:** Confirm the handler no longer constructs regexes directly with `new RegExp(pattern, flags)`.
   - **Stop if:** The shared regex helper from task `1.3` does not match the assumed API.
2. **Add candidate-byte and result-count budget enforcement during traversal**
   - **What:** Track `totalBytesScanned` and enforce candidate-byte/runtime budgets while iterating files and matches.
   - **Where:** `src/domain/inspection/search-file-contents-by-regex/handler.ts`
   - **How:** Use file metadata/size before `fs.readFile` and call `assertRegexRuntimeBudget(...)` after each collected location.
   - **Why:** The endpoint must stop before scanning arbitrarily large content sets or collecting too many locations.
   - **Inputs / References:** shared preflight helper, shared regex runtime helper, shared limits.
   - **Output Contract:** The runtime budget is enforced before and during content scanning.
   - **Verification:** Confirm the handler now tracks candidate bytes and enforces a maximum of `REGEX_SEARCH_MAX_CANDIDATE_BYTES`.
   - **Stop if:** The handler would need a new request property to enforce the runtime budget.
3. **Replace full-line text with bounded excerpts and final family-cap enforcement**
   - **What:** Normalize reported line content through the shared excerpt helper and enforce the final regex-family output cap before returning success.
   - **Where:** `src/domain/inspection/search-file-contents-by-regex/handler.ts`
   - **How:** Use `normalizeRegexMatchExcerpt()` for every returned `content` field and `assertActualTextBudget()` on the final formatted output.
   - **Why:** This controls response size without semantically banning legitimate searches.
   - **Inputs / References:** shared regex helper, shared text-budget helper, shared limits.
   - **Output Contract:** Reported line excerpts are bounded and final text output respects the regex-search family cap.
   - **Verification:** Confirm returned line content is excerpted rather than full trimmed lines and that the final formatted output is budget-checked.
   - **Stop if:** Implementing excerpt shaping would require removing existing `content` or `match` fields from the structured result.
