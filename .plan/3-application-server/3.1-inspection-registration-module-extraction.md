# Task 3.1: Inspection registration module extraction
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]
Create a dedicated application-layer registration module for the inspection bounded context.
The new module must compose domain-owned handler and schema surfaces for all inspection tools while keeping the application layer as a thin MCP registration surface.

## Expected Result
[INTENT: KONTEXT]
- A new file [`register-inspection-tool-catalog.ts`](src/application/server/register-inspection-tool-catalog.ts) exists.
- The file registers all inspection tools currently handled in the monolithic catalog except the server-scope tool `list_allowed_directories`.
- Domain-owned output schemas are imported from the inspection schema modules instead of being re-declared locally.

## Preconditions
[INTENT: CONSTRAINT]
- Plan dependency `D1` is resolved, meaning Unit 1 has completed and the inspection domain handoff surface is stable.
- If [`tool-registration-presets.ts`](src/application/server/tool-registration-presets.ts) exists because task `3.3` was already executed, it must be used as the preset source.

## Context
[INTENT: KONTEXT]
- **Domain:** Application Server / Inspection composition
- **Related Files:** `src/application/server/register-tool-catalog.ts`, `src/application/server/filesystem-server.ts`, all inspection handler/schema modules, optional future `src/application/server/tool-registration-presets.ts`
- **Technical Context:** Extracting one bounded application registration module from a monolithic MCP catalog without changing the registered tool behavior.
- **Business Context:** The application layer must stay thin while the inspection bounded context provides the canonical contract surfaces. This improves modularity and keeps the MCP transport shell scalable.
- **Architectural Context:** Under DDD and 12-Factor, the application layer orchestrates transport wiring and imports canonical domain contracts. It must not own domain result schemas or mix all tool families in one monolithic file.
- **Plan Context Dependencies:** The inspection domain must already be complete (`PLAN:D1`).
- **Runtime Discovery Context:** None. The inspection tool set and their source modules are already known.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Inspection registration module boundary
**Type:** `object`

**Description:**
The new inspection registration module encapsulates the registration of all inspection tools currently embedded in the monolithic catalog.

**Ist-Zustand:**
[`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) registers inspection, comparison, mutation, and server-scope tools in one file.

**Soll-Zustand:**
[`register-inspection-tool-catalog.ts`](src/application/server/register-inspection-tool-catalog.ts) exports a dedicated registration function that owns only the inspection tool family.

**Architectural Rationale:**
This is an application-layer composition split, not a domain move. The goal is to mirror bounded contexts at the registration boundary and shrink the root composition file.

**Exact Target:**
`src/application/server/register-inspection-tool-catalog.ts`

**Implementation Contract:**
The new module must register these tools and no others: `read_files_with_line_numbers`, `list_directory_entries`, `find_paths_by_name`, `find_files_by_glob`, `search_file_contents_by_regex`, `count_lines`, `get_file_checksums`, `verify_file_checksums`, `get_path_metadata`.

**Scenario-Specific Specification:**
| Field | Value |
|------|-------|
| Target Artifact | `registerInspectionToolCatalog` |
| Full Target Shape / Value | `Application registration module containing only the nine inspection tool registrations and the shared registration context contract` |
| Authoritative Source | Current inspection registrations in [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) plus the stabilized inspection domain schemas |
| Determination Rule | Copy behavior, move ownership, and import domain-owned output schemas; do not change tool registration semantics |
| Exact Placement | `src/application/server/register-inspection-tool-catalog.ts` |
| Consumers | Final composition root in task `3.4` |
| No-Breaking Constraint | Same tool names, descriptions, handlers, annotations, execution hints, and output-schema semantics |

## Actions
[INTENT: ANWEISUNG]
1. **Reconfirm the inspection tool set and registration behavior**
   - **What:** Re-read the monolithic application catalog and the stabilized inspection domain modules.
   - **Where:** [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) plus all inspection schema references above.
   - **How:** Extract the exact inspection registration block, including tool names, descriptions, annotations, execution hints, and structured output schema usage.
   - **Why:** The extracted module must preserve behavior exactly.
   - **Inputs / References:** All authoritative references listed above.
   - **Output Contract:** A verified inspection registration blueprint for the new module.
   - **Verification:** The blueprint covers all nine inspection tools and excludes non-inspection tools.
   - **Stop if:** The inspection domain is not yet stable or the existing registration behavior differs from the assumptions in this task file.
2. **Create the dedicated inspection registration module**
   - **What:** Create [`register-inspection-tool-catalog.ts`](src/application/server/register-inspection-tool-catalog.ts).
   - **Where:** `src/application/server/`
   - **How:** Implement one exported registration function that accepts the same registration context shape used by the monolithic catalog, registers only the inspection tools, imports domain-owned output schemas, and imports shared presets from [`tool-registration-presets.ts`](src/application/server/tool-registration-presets.ts) if that file exists.
   - **Why:** This is the first bounded extraction that shrinks the monolithic application composition surface.
   - **Inputs / References:** Current monolithic catalog, inspection domain modules, optional preset module.
   - **Output Contract:** The new module is behaviorally equivalent for the inspection tool family and contains no local result-schema ownership.
   - **Verification:** The file registers only inspection tools and imports output schemas from the inspection domain.
   - **Stop if:** The preset module is missing or the extraction would require redefining domain contracts locally.
3. **Leave root composition unchanged in this task**
   - **What:** Keep [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts) and [`filesystem-server.ts`](src/application/server/filesystem-server.ts) read-only.
   - **Where:** Those two application files
   - **How:** Use them as behavior references only.
   - **Why:** Final root composition rewiring belongs to task `3.4`.
   - **Inputs / References:** [`register-tool-catalog.ts`](src/application/server/register-tool-catalog.ts), [`filesystem-server.ts`](src/application/server/filesystem-server.ts)
   - **Output Contract:** No existing application file is modified by this task.
   - **Verification:** Only the new module file is created.
   - **Stop if:** Extraction appears to require immediate root rewiring.
