### IU-001: Exact Schema Changes for `SearchFileContentsByRegexArgsSchema`
**Type:** `schema`

**Description:**
Apply the exact shared finite limits to the regex search request surface.

**Ist-Zustand:**
The schema leaves `roots`, `regex`, and the glob arrays effectively unbounded and sets `maxResults` only through a default, not a hard maximum.

**Soll-Zustand:**
The schema MUST change exactly as follows:

| Field | Value |
|------|-------|
| `roots` | `z.array(z.string().max(PATH_MAX_CHARS)).min(1).max(MAX_REGEX_ROOTS_PER_REQUEST)` |
| `regex` | `z.string().min(1).max(REGEX_PATTERN_MAX_CHARS)` |
| `includeGlobs` | `z.array(z.string().max(GLOB_PATTERN_MAX_CHARS)).optional().default([]).max(MAX_INCLUDE_GLOBS_PER_REQUEST)` |
| `excludeGlobs` | `z.array(z.string().max(GLOB_PATTERN_MAX_CHARS)).optional().default([]).max(MAX_EXCLUDE_GLOBS_PER_REQUEST)` |
| `maxResults` | `z.number().int().min(1).max(REGEX_SEARCH_MAX_RESULTS_HARD_CAP).optional().default(100)` |
| `caseSensitive` | unchanged |

**Architectural Rationale:**
This gives the endpoint a finite, caller-visible request surface while keeping legitimate regex expressiveness intact. Runtime content semantics remain in task `2.5`.

**Exact Target:**
`src/domain/inspection/search-file-contents-by-regex/schema.ts`

**Implementation Contract:**
Do not add new schema fields. Keep names, defaults, and optionality unchanged except for the new finite caps and the `regex.min(1)` rule.
