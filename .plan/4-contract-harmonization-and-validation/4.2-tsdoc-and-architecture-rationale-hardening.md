## Context
[INTENT: KONTEXT]
- **Domain:** In-repo architectural explanation for the new guardrail system.
- **Related Files:** All modified server, shared, inspection, comparison, and mutation files.
- **Technical Context:** The codebase currently has mixed documentation depth. The new guardrail rollout introduces multiple new concepts—shared limits, metadata-first admission control, structural regex rejects, cumulative text budgets, and a final server fuse—that must be documented clearly.
- **Business Context:** Future maintainers need to understand why the server uses a hybrid multi-layer guardrail system instead of a broad regex blacklist or a schema-only approach.
- **Architectural Context:** Documentation must explain the layered design truthfully, in English, without migration/provenance text and without unsafe doc-comment fragments such as raw `*/`.

## Information Units
[INTENT: SPEZIFIKATION]

### IU-001: Exact Documentation Themes That Must Exist
**Type:** `constraint`

**Description:**
Define the exact rationale themes that the new or updated TSDoc must cover.

**Ist-Zustand:**
The current codebase does not document the new multi-layer guardrail architecture because it does not exist yet.

**Soll-Zustand:**
Documentation added or updated by this task MUST explicitly explain, where relevant:
- why schema caps are the primary contract layer for statically expressible limits
- why metadata-first handler preflights exist for file-size-sensitive endpoints
- why regex runtime safety uses a tiny structural reject layer plus budgets instead of a broad blacklist
- why the global server-shell fuse exists only as the last safety layer
- why hard caps are not caller-overridable

**Architectural Rationale:**
These themes preserve the intent of the rollout and help prevent future regressions back into brittle or duplicated guardrail patterns.

### IU-002: Exact Documentation Rules and Prohibitions
**Type:** `constraint`

**Description:**
Define the exact constraints the documentation update must follow.

**Ist-Zustand:**
No rollout-specific documentation rules are currently enforced in the affected files.

**Soll-Zustand:**
All updated docs MUST follow these rules:
- English only for business/runtime content
- No provenance, migration, or “previously/legacy” wording
- No raw `*/` substrings inside doc comments
- No claims that a caller can disable hard caps
- No placeholder text or incomplete rationale stubs

**Architectural Rationale:**
The documentation must remain production-grade and syntactically safe while reinforcing the new architectural constraints accurately.

## Actions
[INTENT: ANWEISUNG]
1. **Add or update TSDoc on shared guardrail modules**
   - **What:** Add complete TSDoc to the new shared guardrail modules and any updated exported members introduced by the rollout.
   - **Where:** `src/domain/shared/guardrails/**/*.ts`
   - **How:** Document purpose, rationale, key parameters/returns, and the specific LLM/misuse failure mode prevented by each shared guardrail surface.
   - **Why:** The shared guardrail modules are the architectural single source of truth and need the clearest explanation.
   - **Inputs / References:** final shared helper implementations, external governance reference.
   - **Output Contract:** Shared guardrail modules have complete, syntactically safe TSDoc aligned to IU-001 and IU-002.
   - **Verification:** Confirm every new exported function/constant group introduced by the rollout has complete TSDoc.
   - **Stop if:** The final shared helper semantics are not yet stable enough to document truthfully.
2. **Add or update endpoint-level rationale docs where the rollout changes behavior materially**
   - **What:** Update affected inspection, comparison, and mutation files with concise TSDoc or remarks where the guardrail behavior is newly important to understand.
   - **Where:** The affected files under `src/domain/inspection`, `src/domain/comparison`, and `src/domain/mutation`.
   - **How:** Document why the endpoint now refuses, preflights, or budget-checks in the specific places where behavior changed materially.
   - **Why:** Maintainers should not have to infer the guardrail architecture only from shared helper modules.
   - **Inputs / References:** final endpoint implementations, shared guardrail docs.
   - **Output Contract:** Materially changed endpoint files contain concise rationale docs without duplicating every shared constant table.
   - **Verification:** Confirm each materially changed endpoint surface has a clear explanation of its new guardrail behavior.
   - **Stop if:** Writing truthful docs would require assuming behavior not actually implemented.
3. **Update server-level rationale docs**
   - **What:** Add or update TSDoc/comments in the application-layer server files where the final fuse and visible contract harmonization changed semantics.
   - **Where:** `src/application/server/**/*.ts`
   - **How:** Document the role of the global fuse and the visible contract harmonization without restating the entire endpoint catalog.
   - **Why:** The server shell is the final enforcement boundary and must be documented accordingly.
   - **Inputs / References:** final server-shell implementation and registration text.
   - **Output Contract:** Application-layer docs accurately describe the final safety-floor role of the server shell.
   - **Verification:** Confirm the docs describe the global fuse as a last-layer floor, not as the only guardrail.
   - **Stop if:** The code does not yet reflect the final intended semantics.
