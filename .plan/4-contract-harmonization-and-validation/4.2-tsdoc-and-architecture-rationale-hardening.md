## Information Units

### IU-001: Exact Documentation Themes That Must Exist
**Type:** `constraint`

**Description:**
Define the exact rationale themes that the new or updated TSDoc must cover.

**Ist-Zustand:**
The current codebase does not document the new multi-layer guardrail architecture because it does not exist yet.

**Soll-Zustand:**
Documentation added or updated by this task MUST explicitly explain, where relevant:
- why schema caps are the primary contract layer for statically expressible limits
- why metadata-first handler preflights exist for file-size-sensitive endpoints
- why regex runtime safety uses a tiny structural reject layer plus budgets instead of a broad blacklist
- why the global server-shell fuse exists only as the last safety layer
- why hard caps are not caller-overridable

**Architectural Rationale:**
These themes preserve the intent of the rollout and help prevent future regressions back into brittle or duplicated guardrail patterns.

### IU-002: Exact Documentation Rules and Prohibitions
**Type:** `constraint`

**Description:**

**Ist-Zustand:**
No rollout-specific documentation rules are currently enforced in the affected files.

**Soll-Zustand:**
All updated docs MUST follow these rules:
- English only for business/runtime content
- No provenance, migration, or “previously/legacy” wording
- No raw `*/` substrings inside doc comments
- No claims that a caller can disable hard caps
- No placeholder text or incomplete rationale stubs

**Architectural Rationale:**
The documentation must remain production-grade and syntactically safe while reinforcing the new architectural constraints accurately.
