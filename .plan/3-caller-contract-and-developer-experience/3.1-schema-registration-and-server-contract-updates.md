### IU-001: Traversal schema and registration descriptions
**Type:** `schema`

**Description:**
Align traversal-based schema descriptions and registration text with the implemented default-exclude and additive override behavior.

**Ist-Zustand:**
Traversal-based schemas mainly describe `excludeGlobs` and current hard caps; registration descriptions do not mention default excluded trees, explicit roots, or optional `.gitignore` participation.

**Soll-Zustand:**
Traversal-based schemas and registration descriptions explain that broad roots exclude default vendor/cache trees by default, explicit roots remain valid, additive re-includes exist, and `.gitignore` participation is optional.

**Architectural Rationale:**
Agents should not need handler-level source reading to understand the traversal contract.

**Exact Target:**
All traversal-based inspection schemas plus [`register-inspection-tool-catalog.ts`](src/application/server/register-inspection-tool-catalog.ts)

**Implementation Contract:**
Keep descriptions concise but explicit; clarify parameter meaning without flooding the tool surface.

**Scenario-Specific Specification:**
| Field | Value |

| Target Artifact | Traversal-based inspection schemas and inspection registration descriptions |
| Full Target Shape / Value | Updated parameter descriptions for default exclusions, additive re-includes, explicit roots, and optional `.gitignore` enrichment |

| Determination Rule | Descriptions must steer agents toward narrow roots and explicit overrides instead of accidental broad-root traversal |

## Actions
[INTENT: ANWEISUNG]
1. **Update all traversal-based schemas additively**
   - **What:** Add the final caller-facing parameter descriptions for default exclusions, additive re-includes, and optional `.gitignore` participation.
   - **Where:** The five traversal-based inspection schemas.
   - **How:** Preserve existing field names and add new optional fields with concise descriptions that distinguish traversal overrides from other filters.
   - **Why:** Agents must understand the new policy without reading handler code.
   - **Inputs / References:** The implemented endpoint behavior from Unit 2.
   - **Output Contract:** Schema text is explicit, concise, and consistent across endpoints.
   - **Verification:** An agent can tell the difference between `includeGlobs` file filters and `includeExcludedGlobs` traversal overrides where both exist.
   - **Stop if:** A description would require inventing behavior not present in the implementation.
2. **Update inspection registration descriptions**
   - **What:** Revise traversal-based tool descriptions in [`register-inspection-tool-catalog.ts`](src/application/server/register-inspection-tool-catalog.ts).
   - **Where:** Inspection tool registration module.
   - **How:** Mention default vendor/cache exclusions under broad roots, explicit excluded-root access, and additive overrides in prompt-efficient language.
   - **Why:** Registration descriptions are often the first contract surface an agent sees.
   - **Inputs / References:** Traversal-based schema descriptions.
   - **Output Contract:** Tool descriptions and schemas agree.
   - **Verification:** No traversal-based tool description still implies caller-only exclusion semantics.
   - **Stop if:** The update would make the descriptions longer than necessary or drift from the schema text.
3. **Update server instructions**
   - **What:** Extend [`server-instructions.ts`](src/application/server/server-instructions.ts) with the new traversal hardening summary.
   - **Where:** Server-level instruction text.
   - **How:** Add concise lines about default excluded trees, explicit roots, additive re-includes, and optional `.gitignore` enrichment.
   - **Why:** Agents need a stable initialization-level summary of the new traversal model.
   - **Inputs / References:** Traversal-based schema and registration updates.
   - **Output Contract:** The server-level instruction surface advertises the new traversal rules.
   - **Verification:** The new lines remain concise and match the implemented behavior.
   - **Stop if:** The instruction text would need to document a behavior that the code does not actually implement.
