# Task 1.11: List-directory-entries end-to-end continuation response surfacing
[INTENT: ANWEISUNG]

## Goal
[INTENT: KONTEXT]

Close the remaining end-to-end architecture gap in which a real MCP client can call `list_directory_entries` on a normal large directory tree and still receive only a compact preview text message with no usable directory-entry payload and no usable continuation token.

The goal of this task is not to redesign the accepted continuation model. The goal is to make the already accepted architecture actually reach the real client boundary.

## Expected Result
[INTENT: KONTEXT]

- A real MCP client calling `list_directory_entries` on a normal large directory tree can obtain usable bounded directory-entry data, a usable `continuationToken` when more output remains, or both.
- `list_directory_entries` no longer degenerates into a text-only dead end for normal large listing workloads that merely exceeded inline delivery.
- Final preview-first without token remains allowed only when the real client-visible response already includes the bounded final payload and no further resume step is needed.
- Existing same-endpoint continuation semantics remain intact.
- Existing response caps and the global response fuse remain unchanged.
- If the remaining failure is proven to be outside this repository, that external ownership boundary is recorded explicitly instead of leaving the in-repository task falsely marked complete.

## Preconditions
[INTENT: CONSTRAINT]

- Task `1.10` must be treated as resolved but insufficient for the new live client evidence.
- The execute agent must preserve the same-endpoint continuation architecture from tasks `1.8` through `1.10`.
- The execute agent must not solve this task by raising caps, weakening the fuse, introducing a second endpoint, or requiring the client to resend query-defining fields.

## Problem Statement
[INTENT: KONTEXT]

The user has now provided live evidence from the actual MCP client path.

The observed call pattern is:

- endpoint: `list_directory_entries`
- one root directory
- recursive traversal enabled
- user-observed workload roughly around `2423` entries

The user-visible result is currently equivalent to:

```text
Directory listing preview is available for 1 root with 2423 top-level entries in this bounded chunk.
The authoritative directory-entry payload remains in structuredContent.
This preview-first response is finalized and exposes no active continuation token.
```

From the real client point of view, this means:

- no visible bounded entry payload,
- no visible continuation token,
- and therefore no usable way to recover the full structure through the same endpoint.

That is still below the target architecture for a normal large directory listing.

## Target Architecture Clarification
[INTENT: KONTEXT]

The accepted target architecture for inspection endpoints remains:

1. **Inline lane** for small responses.
2. **Preview-first or task-backed lane** when inline delivery is no longer valid.
3. **Same-endpoint continuation** through `continuationToken` when more output remains after the current delivered chunk.
4. **Later hard-gap refusal** only when even bounded continuation-style delivery is no longer valid or safe.

For `list_directory_entries` specifically, the architectural end state is:

- it may still be a preview-capable family,
- but preview-first must remain useful to the real client,
- and the real client must not be forced to request deeper and deeper subpaths merely because a normal large listing exceeded inline response boundaries.

## Architectural Clarifications Bound by This Task
[INTENT: KONTEXT]

### A. What the architecture does **not** mean

It does **not** mean:

- every preview-first response must always carry a token,
- or every large response must always be fully returned in one call,
- or the client can bypass all hard gaps just by asking again.

### B. What the architecture **does** mean

It **does** mean:

- a normal large discovery workload must not collapse into a useless stop state,
- the client must receive a usable bounded payload and/or a usable continuation path,
- and the server cannot claim architectural success merely because an internal structured contract exists while the real client still cannot consume the result.

### C. Final preview-first without token

Final preview-first without token is architecturally valid **only** when:

- no further resume step exists,
- and the bounded final payload of that last preview response actually reaches the client.

If the client only sees compact text and no payload and no token, then the end-to-end contract is still incomplete.

## Likely Error Sources
[INTENT: KONTEXT]

The most likely fault domains are now:

1. **Server-side premature finalization**
   - `list_directory_entries` may still be finalizing the response as non-resumable too early for the actual delivered client surface.

2. **In-repository response-shell or tool-registration surfacing loss**
   - an in-repository layer may still be collapsing the authoritative structured payload before the real MCP client sees it.

3. **External consumer boundary**
   - if the repository emits the correct structured payload and token-bearing metadata, but a downstream MCP client or adapter outside this repository drops it, that external boundary must be stated explicitly.

The execute agent must prove which of these is true.

## Architectural Non-Goals
[INTENT: KONTEXT]

This task must **not**:

- re-open the already accepted same-endpoint continuation model as such,
- convert `list_directory_entries` into a second public pagination API,
- make the client resend `roots` during resume,
- raise response caps,
- weaken the global fuse,
- or cancel preview-first for all inspection families.
